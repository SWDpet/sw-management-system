# SW Manager 시스템 기능 정의서

> **최종 수정일:** 2026-03-17
> **목적:** 시스템 전체 기능을 정의하고, 수정 시 기존 기능 보호를 위한 절대 지침을 수립한다.

---

## 1. 시스템 개요

| 항목 | 내용 |
|------|------|
| 시스템명 | SW Manager |
| 기술스택 | Spring Boot + Thymeleaf + JPA + PostgreSQL |
| 인증방식 | Spring Security (세션 기반) + CSRF 토큰 |
| DDL 정책 | `ddl-auto=none` (수동 마이그레이션) |

---

## 2. 권한 체계

| 권한등급 | 설명 |
|---------|------|
| ROLE_ADMIN | 전체 모듈 접근 + 사용자 관리 |
| NONE | 해당 모듈 접근 불가 |
| VIEW | 목록 조회, 상세 보기, 출력 |
| EDIT | VIEW 권한 + 등록, 수정, 삭제 |

### 모듈별 권한 필드

|------|----------|
| authDashboard | 대시보드 |
| authProject | 사업현황 |
| authPerson | 인적정보 |
| authInfra | 서버관리 |
| authLicense | 라이선스 |
| authQuotation | 견적서 |

## 4. 공통 기술 규칙

### 4.1 CSRF 토큰
- 모든 POST/PUT/DELETE 요청에 CSRF 토큰 필수
- AJAX 요청: `fetchWithCsrf()` 래퍼 함수 사용
- 동적 폼: `<input type="hidden" name="_csrf">` 추가 필수

### 4.2 DB 마이그레이션
- `ddl-auto=none` 정책: 스키마 변경은 반드시 SQL 파일 작성
- 위치: `swdept/sql/V0XX_설명.sql`
- Entity 변경 시 반드시 대응하는 SQL 파일 동반

### 4.3 검색
- 단일 키워드: 모든 주요 필드에 LIKE 검색
- AND 검색: `&` 구분자 (예: `2024&진행중&UPIS`)
- 계약상태 검색: 상태명 → 코드 자동변환

### 4.4 페이징
- 기본 10건/페이지 (관리자 로그는 20건)
- Pageable + Sort 조합

### 4.5 정렬 기준

| 모듈 | 정렬 기준 |
|------|----------|
| 사업현황 | 년도 DESC → 시도명 ASC → 시군구명 ASC → 시스템명 ASC |
| 인적정보 | 시도 → 시군구 → 기관 → 시스템명 ASC |
| 서버관리 | infraId DESC |
| 견적서 | 최신순 |
| 로그 | 시간 DESC |

## 5. ⚠️ 수정 시 절대 지침

> **이 지침은 코드 수정 시 반드시 준수해야 하는 절대적 규칙이다.**

### 지침 1: 기존 기능 보호 원칙
- **수정 전 반드시 이 문서를 확인**하여 해당 모듈의 기존 기능 목록을 파악한다.
- 수정 대상 파일의 **모든 기존 기능이 정상 동작하는지 확인**한 후 수정에 착수한다.
- 한 기능을 추가/수정할 때 **같은 파일 내 다른 기능을 절대 손상시키지 않는다.**

### 지침 2: 영향 범위 분석 필수
- 수정할 파일을 공유하는 다른 페이지/기능이 있는지 반드시 확인한다.
- 예시: `QuotationController.java` 수정 시 → 목록, 상세, 미리보기, 관리대장, 패턴, 노임단가 등 모든 관련 기능 영향 체크
- 공통 서비스(`SwService`, `QuotationService` 등) 수정 시 → 해당 서비스를 참조하는 모든 컨트롤러 확인

### 지침 3: CSRF 토큰 누락 방지
- POST/PUT/DELETE 요청을 추가할 때 반드시 CSRF 토큰 포함 여부를 확인한다.
- 동적 폼 생성 시 `_csrf` hidden input 또는 헤더 전달 누락 금지.
- **위반 시 403 Forbidden 발생.**

### 지침 4: DB 스키마 변경 시 SQL 동반
- Entity 컬럼 추가/변경/삭제 시 반드시 `swdept/sql/` 에 마이그레이션 SQL 작성.
- `columnDefinition` 변경 시에도 SQL 필수.
- 기존 데이터 무결성을 해치는 변경 금지.

### 지침 5: 권한 체크 일관성 유지
- 새 엔드포인트 추가 시 반드시 `checkViewAuth()` 또는 `checkEditAuth()` 적용.
- 조회 기능: `checkViewAuth()`, 생성/수정/삭제: `checkEditAuth()`
- VIEW 권한 사용자에게 EDIT 기능 노출 금지 (HTML에서 `th:if="${canEdit}"` 사용).

### 지침 6: Thymeleaf 인라인 JavaScript 규칙
- `th:inline="javascript"` 블록 내에서만 Thymeleaf 변수 사용.
- 일반 `<script>` 태그에서 `[[${...}]]` 사용 금지 (렌더링 안 됨).

### 지침 7: 견적서 금액 계산 규칙 보호
- 절사(ROUNDDOWN) 규칙: 별도 지침 `절사_규칙_가이드.md` 참조.
- 금액 계산 흐름: rawTotal → VAT → 절사(합계) → 낙찰율 → 절사 → grandTotal
- 이 흐름을 변경하거나 순서를 바꾸지 않는다.
- 수량/단가/금액 0일 경우 견적서 양식에 빈칸 표시 (0 출력 금지).

### 지침 8: 정렬 기준 보호
- 각 모듈의 정렬 기준은 본 문서 §4.5에 정의되어 있다.
- 정렬 기준 변경 시 반드시 이 문서와 해당 별도 지침 MD를 함께 수정한다.
- 사업현황 정렬 기준: 별도 지침 `사업현황_정렬_기준.md` 참조.

### 지침 9: 수정 후 검증 체크리스트
수정 완료 후 다음 항목을 반드시 확인한다:

- [ ] 수정한 파일의 기존 기능이 모두 정상 동작하는가?
- [ ] CSRF 토큰이 필요한 요청에 모두 포함되어 있는가?
- [ ] 권한 체크가 누락된 엔드포인트는 없는가?
- [ ] DB 스키마 변경이 있다면 SQL 마이그레이션이 작성되었는가?
- [ ] 공유 파일을 사용하는 다른 기능에 영향이 없는가?
- [ ] 삭제된 함수/변수를 참조하는 다른 코드가 없는가?
- [ ] 인라인 미리보기와 출력 미리보기 양쪽 모두 동일하게 반영되었는가?

### 지침 10: 문서 업데이트 의무
- 새 기능 추가 또는 기존 기능 변경 시 **이 문서를 반드시 업데이트**한다.
- 별도 규칙 MD 파일이 있는 경우 해당 파일도 함께 수정한다.
