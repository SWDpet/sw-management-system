## 0. 철학

- 목표는 **"완벽한 코드"가 아니라 "싸고 안전하게 진화하는 코드"**.
- 토대는 건강하다(Spring Boot·JPA·계층분리·Security·SQL 파라미터 바인딩=A). 문제는 **초기 관례부채**(무타입 Map 남용·응답 무표준·거대 클래스)뿐. → **전면 리팩토링 아님**, 점진 상환.
- **속도 < 품질**: 증분이 느려도 검증·게이트·이중검수(codex) 충실히.

## 1. 등급 정의 (C/B/A + S 신설)

| 등급 | 정의 |
|------|------|
| 🔴 C | 감사 지적 3+ 또는 하드코딩 다수 |
| 🟡 B | 감사 지적 1~2건 또는 매직넘버·리터럴 일부 잔존 |
| 🟢 A | **감사 지적 0** · 테스트 커버리지 있음 · Enum/Master 활용 |
| 🔷 **S (beyond-A)** | A 충족 **+ 아래 §3 S-기준을 *게이트로 강제*** (스냅샷이 아니라 *추세·자동화*로 측정) |

## 2. 감사 범위 — "무엇이 지적인가" (먼저 고정)

코드품질/아키텍처 등급을 매길 때 **지적으로 카운트하는 항목**(이 목록 밖은 등급에 반영 안 함):

| 코드 | 지적 항목 | 측정 |
|---|---|---|
| F1 | **공개 API의 무타입 `Map<String,Object>` 남용** (요청 바디·응답 envelope) | `rg` 카운트, baseline 대비 |
| F2 | **거대 클래스** (라인 임계 초과) | 임계 **분리(codex)**: 컨트롤러 **1,500줄** / 서비스 **2,000줄**. 초과 클래스 수 |
| F3 | **매직넘버·리터럴** (상수/Enum 미사용) | 핫스팟 파일 한정 검토 |
| F4 | **권한 가드 누락** 변이 엔드포인트 | `rg` + 회귀 테스트 |
| F5 | **문서-코드 drift** (stale 좌표·형태 불일치) | living 문서 한정 |
| F6 | **`@SuppressWarnings("unchecked")`** 밀집 | 카운트, 감소 추세 |
| F7 | **응답 무표준** (엔드포인트마다 제각각 envelope: error 맵/문자열·data 변덕) | ApiResult 적용률 |
| F8 | **책임 누수** (컨트롤러에 비즈니스 로직·트랜잭션 경계 누수) | 핫스팟 리뷰 |
| F9 | **테스트 없는 보안·권한 변경** (가드/CSRF 변경에 회귀 테스트 부재) | PR 단위 점검 |

> 정직성 원칙: F1·F2·F7 같은 "진짜 냄새"를 범위에서 빼고 A를 선언하지 않는다.

## 3. S-기준 (A 위에 추가로 충족, *게이트로* 강제)

| 코드 | S 기준 | 현재 |
|---|---|---|
| S1 | **부채 무증가 게이트**: 신규 무타입 Map(F1)·거대클래스(F2) 0, 기존은 baseline+**감소만 허용** | ⏳ 미도입 |
| S2 | **테스트 신뢰** (codex 판정식 고정): ① **기준 대비 커버리지 하락 금지**(JaCoCo, com.swmanager) ② 핵심 순수로직 패키지(예: `service`·`response`)에 **PIT 뮤테이션 최소 점수**(잠정 60%) | 하락금지 ✅(ratchet) / PIT ⏳ |
| S3 | **격리 테스트**: 운영DB 비의존 (Testcontainers 등). ⚠ 현재 Docker 미설치 제약 → *조건부*(Docker 도입 시) | ⏳ (Docker 필요) |
| S4 | **운영성**: 기동 fail-fast + **구조적 로깅 정의** = 에러는 코드/컨텍스트 키 포함 구조화 로그, 민감값 미포함 | 기동 fail-fast ✅ (DbInitRunner) |
| S5 | **문서-코드 정합 0 drift**: **living 문서 목록 고정** = `QUALITY_*`·`SECURITY.md`·`RELIABILITY.md`·`DB_CONNECTION.md`·`ARCHITECTURE.md`. 이들만 상시 0 drift | ✅ (S2 스프린트) |
| S6 | **자동 게이트가 빌드를 막는다** (수동 점검 아님) | JaCoCo만 ✅, 나머지 ⏳ |

## 4. 운영 원칙 — ratchet + hotspot + 출혈중단

1. **출혈 중단**: *새* 코드에 부채 금지 (S1 게이트). 더 나빠지지 않게 먼저.
2. **핫스팟만 정조준**: 자주 바뀌는·버그 나는·등급 막는 곳만. 현재 핫스팟(codex 식별):
   - DocumentController(Map 다수), **OpsDocController(다음 거대클래스 분리 대상)**, QuotationController(`ResponseEntity<Map>` 반복 → ApiResult).
3. **안정적·미사용 "못난 코드"는 둔다**: 전면 리팩토링은 *이득 없이 위험만* (음의 ROI).
4. **baseline + 감소만**: 기존 부채는 baseline 기록 → 증가 시 빌드 실패, 감소는 OK.
5. **범위 먼저 문서화 후 재채점** (자기합리화 차단).
6. 모든 변경은 **codex 이중검수 + 게이트 통과** 후 커밋.

## 5. 게이트 카탈로그 (도입 순서)

| 게이트 | 강제 대상 | 상태 |
|---|---|---|
| JaCoCo ratchet | 커버리지 하한(com.swmanager) | ✅ 도입(S3) |
| **Map baseline 게이트** | F1 신규 0 / 총량 감소만 | ⏳ 다음 |
| **ArchUnit 컨트롤러 라인** | F2 (>1,500줄 클래스 추가 금지) | ⏳ |
| rg 신규금지 | 신규 `@RequestBody Map`·`ResponseEntity<Map<String,Object>>` | ⏳ |
| Enum/Master sync | 마스터 drift | ✅(arch 테스트) |

**ratchet 구현 함정 (codex):** ① baseline 파일 **갱신 절차/권한**을 명문화(임의 상향 금지, 감소 시에만 PR로 갱신) ② **생성 코드 제외**(target/, assembly 산출물) ③ `rg` **오탐 최소화**(주석·문자열 내 패턴 배제) ④ **라인 수 측정 방식 고정**(빈줄·주석 포함 여부 등 한 가지로).

## 6. 진행 순서 (현실적)

1. **본 헌장 확정**(codex 검토) — *기준 고정*.
2. **문서 A 확정**: QUALITY_SCORE 재채점(증거 기반) + 잔여 drift 0.
3. **게이트 도입**: Map baseline + ArchUnit 라인(S1/S6 핵심).
4. **핫스팟 정조준**: OpsDocController 분리 → Quotation ApiResult → Document Map 감축.
5. 각 단계 후 **재채점 + 게이트가 등급 고정**.
