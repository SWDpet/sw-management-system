## 0. 철학 (v0.3 전환)

- 목표는 **"깨끗하고 *계속 유지되는* 코드"** — 2026-06-20 사용자 결정: "다 갈아엎더라도 안정적이게 만들어 유지". 단 *지금이 적기*인 이유 = 31k LOC·3개월·**토대가 건강**(Spring·JPA·계층·SQL 바인딩=A)할 때가 가장 싸다.
- **"갈아엎다" ≠ 처음부터 재작성.** 토대는 유지하고, **전 영역 관례부채를 점진 정리**. 정리 대상은 *유지비를 계속 만드는 축*으로 한정(codex): `Map→DTO`·거대 클래스·응답 표준·권한/보안 경계. (= "모든 못난 코드 제거"가 아님)
- **방법 = strangler(점진 재작성), big-bang 금지.** 작동하는 시스템의 일괄 동시 재작성은 실패율이 높다. 모듈 단위로 **net 선행 → 정리 → 게이트**. 항상 배포 가능 상태 유지.
- **net 선행 원칙(사용자 통찰 "되던 게 안 되는 걸 잡는다")**: 안정적 코드도 **characterization/golden net 을 먼저 깐 뒤** 고치면 안전. net 없는 리팩토링 금지(net 없는 변경이 위험한 것이지, 안정 코드를 못 고치는 게 아님).
- **속도 < 품질**: 증분이 느려도 net·게이트·이중검수(codex) 충실히. 이는 **다세션 프로그램**이며 기능 속도 둔화를 수용한다.
- ⚠ **이 저장소 특유 위험(codex)**: 컨트롤러가 URL·JSON·권한·로그·Thymeleaf 흐름을 함께 쥐어, 단순 분리도 프론트 fetch·응답 shape 를 깨기 쉽다 → **net 이 필수**.

## 1. 등급 정의 (C/B/A + S 신설)

| 등급 | 정의 |
|------|------|
| 🔴 C | 감사 지적 3+ 또는 하드코딩 다수 |

| F5 | **문서-코드 drift** (stale 좌표·형태 불일치) | living 문서 한정 |

| F7 | **응답 무표준** (엔드포인트마다 제각각 envelope: error 맵/문자열·data 변덕) | ApiResult 적용률 |
| F8 | **책임 누수** (컨트롤러에 비즈니스 로직·트랜잭션 경계 누수) | 핫스팟 리뷰 |

## 4. 운영 원칙 — net 선행 strangler + ratchet (v0.3)

1. **net 선행(필수)**: 모듈을 고치기 전 그 **현재 동작을 박제**(characterization/golden). "되던 것"을 기록해야 "안 되는 것"을 잡는다. → §4-1 net 인프라.
2. **모듈 단위 strangler**: net → 정리 → 게이트 → 다음 모듈. **big-bang(동시 전면 재작성) 금지.** 매 단계 배포 가능.
3. **출혈 중단**: *새* 코드엔 부채 0 (rg 게이트). 더 나빠지지 않게 먼저.
4. **전 영역 점진(구 '핫스팟만'에서 확대)**: 유지비 축(Map→DTO·거대클래스·응답표준·권한경계)을 **전부** 정리하되 *순서대로*(§6). 진입 순서 우선: OpsDocController·DocumentController·QuotationController.
5. **baseline + 감소만**: 기존 부채 baseline → 증가 시 빌드 실패, 감소 OK.
6. **범위 먼저 문서화 후 재채점** (자기합리화 차단).
7. 모든 변경은 **net 통과 + codex 이중검수 + 게이트 통과** 후 커밋.

### 4-1. net 인프라 (codex — 모듈별 net 을 싸게, Docker 없이)
1. **MockMvc 엔드포인트 골든 스냅샷** (`@SpringBootTest` 남발보다 가벼움)
2. **JSON canonicalizer**: 필드 정렬 + 휘발값(타임스탬프·생성ID) 제거 → 스냅샷 안정화
3. **URL/메서드/응답형태 인벤토리 diff** (분리 전후 1:1 대조)

### 4-2. net 의 한계·위험 (정직)
- **골든은 *현재 버그까지* 고정**한다 → net 작성 시 "이게 올바른 동작인가" 한 번 의심.
- **샘플 데이터 부족** → 분기 커버 불완전.
- **바이너리 출력(PDF·엑셀·HWPX) 회귀** → 골든이 어려움. 구조적 단언(셀/키)로 보완(이미 ExcelExport 적용).

## 5. 게이트 카탈로그 (도입 순서)

| 게이트 | 강제 대상 | 상태 |
|---|---|---|
| JaCoCo ratchet | 커버리지 하한(com.swmanager) | ✅ 도입(S3) |
| **Map baseline 게이트** | F1 신규 0 / 총량 감소만 | ⏳ 다음 |
| **ArchUnit 컨트롤러 라인** | F2 (>1,500줄 클래스 추가 금지) | ⏳ |
| rg 신규금지 | 신규 `@RequestBody Map`·`ResponseEntity<Map<String,Object>>` | ⏳ |
| Enum/Master sync | 마스터 drift | ✅(arch 테스트) |

**ratchet 구현 함정 (codex):** ① baseline 파일 **갱신 절차/권한**을 명문화(임의 상향 금지, 감소 시에만 PR로 갱신) ② **생성 코드 제외**(target/, assembly 산출물) ③ `rg` **오탐 최소화**(주석·문자열 내 패턴 배제) ④ **라인 수 측정 방식 고정**(빈줄·주석 포함 여부 등 한 가지로).

## 6. 진행 순서 (v0.3 — codex 권장 순서)

1. ✅ **본 헌장 확정**(codex 검토) — 기준 고정.
2. **net 인프라 구축**(§4-1): MockMvc 골든 + JSON canonicalizer + 인벤토리 diff. *이게 선행돼야 안전하게 전면 정리 가능.*
3. **문서 A 확정 + 게이트 1차**: QUALITY_SCORE 재채점 + 잔여 drift 0 + Map baseline·ArchUnit 라인 게이트.
4. **응답 표준 통일**: ApiResult 전 컨트롤러 확산(net 보호 하).
5. **거대 컨트롤러 분리**: OpsDocController → DocumentController (net 보호 하 strangler).
6. **서비스 책임 분리**: ExcelExportService 등.
7. **요청 DTO 바인딩**: `@RequestBody Map` → 타입 DTO.
8. 각 단계 후 **재채점 + 게이트가 등급 고정**.

> 이전 §6(헌장 확정→문서→게이트→핫스팟)에서 **net 인프라를 2번으로 끌어올림** — 전 영역 정리의 안전 전제이기 때문.
