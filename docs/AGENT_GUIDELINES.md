# AGENT_GUIDELINES.md — 코딩 에이전트 행동 지침

> 모든 에이전트(Claude Code, codex CLI 등) 공통 코딩 휴리스틱.
> 팀 워크플로우/자동화 룰은 `AGENTS.md`, Claude Code 전용 컨텍스트는 `CLAUDE.md` 참조.

**원칙 vs 속도**: 본 지침은 신중함을 우선한다. 단순 작업은 판단껏 생략 가능.

기반: [andrej-karpathy-skills/AGENTS.md](https://github.com/datajuny/andrej-karpathy-skills) 13원칙.
본 프로젝트 룰과 충돌하는 §6/§8/§9/§10/§12 는 재작성됨.

---

## 1. 코딩 전에 먼저 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 표면화할 것.**

구현 전:
- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여러 개면 모두 제시한다. 임의 선택 금지.
- 더 단순한 접근이 있으면 말한다. 필요하면 사용자 의견에 반론한다.
- 불분명하면 멈춘다. 무엇이 헷갈리는지 명명한다. 묻는다.

## 2. 단순함 우선

**문제를 푸는 최소 코드. 추측성 코드 금지.**

- 요청 범위 밖의 기능 추가 금지.
- 1회용 코드에 추상화 금지.
- 요구되지 않은 "유연성"·"설정 가능성" 추가 금지.
- 발생 불가능한 시나리오를 위한 에러 핸들링 금지.
- 200줄 짠 게 50줄로 가능하면 다시 쓴다.

자가 점검: "시니어가 보면 과한가?" 라면 단순화한다.

## 3. 외과적 변경

**필요한 곳만 손댄다. 본인이 만든 흔적만 정리한다.**

기존 코드 편집 시:
- 인접 코드/주석/포매팅을 "개선" 하지 않는다.
- 망가지지 않은 것을 리팩터하지 않는다.
- 본인 스타일과 달라도 기존 스타일을 따른다.
- 무관한 dead code 발견 시 언급하되 삭제는 X.

본인 변경으로 고아가 된 import/변수/함수만 제거한다. 기존 dead code 는 요청 없이 삭제 X.

검증: 변경된 모든 줄이 사용자 요청에 직접 추적 가능한가.

## 4. 목표 기반 실행

**검증 가능한 성공 기준 정의 후 루프.**

작업을 검증 가능한 목표로 변환:
- "검증 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시킨다"
- "버그 수정" → "재현 테스트 작성 후 통과시킨다"
- "X 리팩터" → "리팩터 전후 테스트 통과 보장"

다단계 작업은 간단한 계획 명시:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
```

강한 성공 기준은 독립 루프를 가능케 한다. 약한 기준("동작하게")은 매번 사용자 확인이 필요하다.

## 5. 편집 전 워크스페이스 증거 확인

**손댈 파일을 직접 읽는다. 기억이나 오래된 요약에 의존 X.**

코드 변경 전:
- `Grep` / `Glob` 으로 관련 구현 위치를 찾는다 (NOT `grep`/`find`).
- 정확한 파일과 인접 호출 지점을 `Read` 로 직접 본다.
- 열린 탭, 파일명, README, 이전 대화 요약은 단서일 뿐 증거 아님.
- 로컬 코드가 가정과 다르면 코드를 신뢰하고 계획을 갱신한다.

"작은 변경" 한정 룰이 아니다. 상상 코드에 기반한 자신만만 편집을 막기 위한 룰.

## 6. 워크트리 존중

**커밋되지 않은 변경은 본인이 만든 게 아니면 사용자 것으로 간주.**

워크트리가 dirty 일 때:
- 무관한 변경을 revert/덮어쓰기/리포맷 X.
- 같은 파일에 사용자 변경이 있으면 읽고 적응한다.
- 무관한 dirty 파일은 무시한다.
- 사용자가 명시 요청하지 않은 destructive git 명령 절대 X.

**본 프로젝트 강화 룰**:
- `git add -A` / `git add .` **절대 금지**. 항상 `git add <명시 파일>`.
- `git reset --hard`, `git checkout --` 같은 destructive 작업은 사용자 명시 승인 후만.
- 자세한 게이트는 `AGENTS.md` §5 "작업완료" 트리거 안전 게이트 참조.

## 7. 한국어 출력 시 마침표 (콜론 X)

**한국어 문장은 마침표로 끝낸다.**

사용자가 한국어로 쓰면 출력도 한국어:
- 다음 줄이 리스트/예시여도 한국어 문장을 `:` 로 끝내지 않는다.
- 영어 문서로 학습된 LLM 은 콜론 습관을 한국어로 흘린다. 잡아낸다.
- 검증: 한국어 문장 종결자는 `.` `?` `!` 중 하나여야 한다.
- 코드, key-value, 라벨 안의 콜론은 OK. 한국어 문장 종결로는 X.

## 8. 신규 파일 한국어 헤더

**모든 신규 소스 파일 첫 줄: 역할을 설명하는 한국어 한 줄 주석.**

본 프로젝트 스택 기준:
- Java: `// 사업 등록 폼 컨트롤러` (package 선언 바로 아래 또는 import 위)
- Thymeleaf: `<!-- 견적서 메인 레이아웃 -->`
- SQL/마이그레이션: `-- 라이선스 만료 알림 큐 테이블`
- JS (소수): `// 견적서 PDF 미리보기 클라이언트 헬퍼`
- Python (HWP 추출 등): `# 한글 파일 텍스트 추출 유틸`
- shebang/지시문이 있으면 그 바로 아래에 둔다.
- 설정 파일 (`*.properties`, `pom.xml`, lockfile, generated) 은 제외.

이유: 에이전트는 파일을 선택적으로 읽는다. 한 줄 한국어 헤더가 다음 세션의 빠른 탐색을 돕는다.

## 9. 기획 + 개발계획 + 결정 노트

**비단순 작업 전 세 산출물을 만든다. 산출물 없이 코딩 시작 X.**

본 프로젝트 매핑:
- **기획서** → `docs/product-specs/<feature>.md` (요건·UI·DB·리스크)
- **개발계획서** → `docs/exec-plans/<feature>.md` (FR/NFR/T 항목 단위 작업 + 검증 기준)
- **결정 노트** → 개발계획서 내 "결정 로그" 섹션 또는 `docs/design-docs/<topic>.md`

각 산출물은 codex 검토 게이트 통과 후 사용자 최종승인. 흐름 전체는 `AGENTS.md` §3.

사용자가 계획만 주고 바로 코딩하라 하면 멈추고 묻는다: "기획/개발계획 산출물 먼저 만들까요." 다음 세션이 모든 결정을 다시 유도하지 않으려면 노트가 필요하다.

## 10. 완료 선언 전 검증

**코드를 만졌으면 "완료" 전 검증을 돌린다.**

본 프로젝트 룰 (AGENTS.md §3 와 정합):
- **테스트 실행은 codex 에 위임**. Claude 가 직접 `mvn test` 실행 X.
- Claude 가 직접 돌릴 수 있는 검증: 빌드/컴파일 (`./mvnw -DskipTests=true compile`), 정적 분석, Grep/Read 기반 영향 분석.
- codex 검증 결과(항목별 충족/미충족)를 사용자에게 그대로 전달.
- 검증을 못 돌릴 상황이면 "왜 못 돌렸는지" 명시.

이 단계는 에이전트가 가장 자주 빠뜨린다. 협상 불가.

## 11. 최종 답변에 검증 증거 포함

**의도한 검증이 아닌, 실제로 돌린 검증을 보고한다.**

최종 응답에는:
- 실행한 명령 (예: `./mvnw -DskipTests=true compile`, `codex review docs/exec-plans/x.md`)
- 결과 (예: "통과", "X 에러로 실패", "Y 사유로 미실행")
- 사용자가 알아야 할 잔여 리스크

"끝", "고침", "동작함" 은 구체적 확인이 뒷받침될 때만 쓴다.

## 12. 의미 단위 커밋

**한 논리 변경이 끝나면 commit 단위로 인식한다. 단, 실제 commit/push 는 게이트에서만.**

본 프로젝트 룰 (AGENTS.md §5 와 정합):
- **자동 commit 금지**. commit/push 는 사용자 "작업완료" 트리거 + confirm 게이트로만 진행.
- Claude 는 작업 중 논리 단위가 끝나면 변경 사항 요약 + commit 후보 메시지를 사용자에게 제시.
- 트리거 발화 시: 변경 echo (`git status`, `git diff --name-only`, `git log --oneline origin/master..HEAD`) → confirm → `git add <명시 파일>` → `git commit -m "의도 중심"` → `git push`.
- 검증: "이 변경을 한 문장으로 설명할 수 있는가." 가능 → 단위 OK. 불가능 → 섞임. 분리.
- 좋은 메시지: "사업 등록 폼 검증 추가". 나쁜 메시지: "사업 등록 + UI + 버그 수정 다 함".
- 금지: `.env`, credential, 대용량 바이너리, 민감 파일.

## 13. 에러를 읽는다, 추측하지 않는다

**실제 에러/로그 줄을 읽는다. 기억으로 패턴 매칭 X.**

실패 시:
- 전체 에러 메시지와 스택 트레이스를 읽는다.
- 가정한 로그가 아닌 실제 출력을 확인한다.
- 원인 확인 전에 "흔한 수정" 적용 금지.
- 불분명하면 print/log 추가로 상태 확인 후 수정.

이 단계는 "테스트 실행" 다음으로 자주 빠뜨린다. 에이전트는 에러 키워드로 최근 본 패턴을 적용한다. 한 줄 버그가 세 파일 리팩터로 변하는 경로.

---

**잘 작동 중인 신호**: diff 의 불필요 변경 감소, 과복잡으로 인한 재작업 감소, 검증 증거가 명령 단위로 보고됨, 의문 질문이 실수 후가 아닌 구현 전에 나옴.
