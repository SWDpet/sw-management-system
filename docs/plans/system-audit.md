## 1. 배경 / 목표

### 배경
- 시스템 관계도 개발 중 **ERD 문서와 실제 코드 간 불일치**(tb_contract 미구현, tb_contract_participant.proj_id 스키마 drift 등) 가 발견됨.
- 그 외에도 누적된 기술 부채(미사용 코드, 민감 정보 노출 가능성, 의존성 노후화 등) 가 있을 수 있으나 체계적으로 점검된 바 없음.
- codex 를 도구로 사용해 **1회성 건강 검진** 형태의 감사를 진행하면 리스크를 조기 발견할 수 있음.

### 목표
- 현재 codebase 에 대해 **제한적이지만 실행 가능한** 카테고리별 감사를 codex 로 수행.
- 발견 사항을 심각도(P1/P2/P3) 별로 정리한 보고서 생성.
- 사용자가 검토 후 **우선순위를 결정**하고, 후속 수정 작업은 별도 스프린트로 분리.
- 본 감사 자체는 **진단·보고서 작성**까지만 — 실제 수정은 범위 밖.

### 비목표 (Phase 2 로 분리)
- 자동 수정 / 자동 PR 생성
- 테스트 커버리지 측정·개선
- 의존성 업그레이드 일괄 적용
- 아키텍처 대대적 리팩토링

## 2. 감사 카테고리 (Phase 1 범위)

| # | 카테고리 | 대상 | 핵심 질문 |
|---|---------|------|----------|
| **C1** | **보안** | `src/main/java`, `application.properties`, `.yml`, `.env*`, `docs/` | 하드코딩된 비밀번호/API 키, 민감 필드 API 노출, @PreAuthorize 누락 엔드포인트 |
| **C2** | **엔티티-DB 스키마 정합성** | `src/main/java/.../domain/`, `src/main/resources/db_init_phase2.sql` | @Column 이름과 init SQL 컬럼 일치, @JoinColumn FK 대상 테이블 존재, 주석·실제 타입 일치 |
| **C3** | **문서 ↔ 구현 일치** | `docs/ERD.md`, `docs/erd-*.mmd`, `docs/plans/`, `docs/dev-plans/` | ERD 가 실제 엔티티와 일치, 계획서의 API 엔드포인트·DTO 필드가 실제로 존재 |
| **C4** | **미사용 코드 (dead code)** | `src/main/java`, `src/main/resources/templates`, `.team-workspace` | private 메서드 중 호출 없는 것, public 메서드 중 참조 0, 템플릿 중 라우트 없는 것 |
| **C5** | **민감 로그** | 전체 서버 로그 출력 포인트 | `log.info` 등에 password, token, 전체 JSON body 출력 여부 |

### 제외 카테고리 (Phase 2)
- 성능 (N+1, slow query)
- 의존성 보안 (CVE 스캔)
- 테스트 커버리지
- 코드 스타일 (checkstyle 등)

## 3. 감사 실행 방식 (워크플로우 내 "개발" 단계에 해당)

### 3-1. 카테고리별 codex 호출
각 카테고리 C1~C5 마다 **1회 이상의 codex review 호출**. 프롬프트 템플릿 (FR-5, FR-6 지시 포함):

```
codex review "우리 codebase 에 대해 [카테고리명] 관점에서 감사해줘.

대상 경로:
- <대상 디렉토리/파일 목록>

핵심 질문:
- <핵심 질문>

결과 형식:
[심각도별 발견]
- P1 (즉시 조치 필요): ...
- P2 (우선 조치 권장): ...
- P3 (선택적 개선): ...

각 발견에는 파일:라인 참조 + 권장 조치 포함.

**민감값 마스킹 (필수, FR-5)**: 발견 위치·유형만 기술하고,
실제 비밀번호/API 키/토큰/인증 값은 반드시 `***` 로 마스킹하여 반환.
예시: `password=secretValue123` 발견 → 보고서에는 `password=***` 로만 기재.

**범위 제외 (FR-6)**: 성능 이슈(N+1, slow query), 의존성 CVE/업그레이드,
테스트 커버리지, 코드 스타일은 이번 감사 범위 밖 — 언급하지 말 것.
이번 감사는 이 카테고리에 국한."
```

### 3-2. 보고서 취합
개발팀(Claude) 이 codex 응답을 받아 **통합 보고서** `docs/audit/{YYYY-MM-DD}-system-audit.md` (FR-4 규칙, YYYY-MM-DD 는 감사 실행일) 로 정리:

```markdown
# 시스템 감사 보고서 (Phase 1)
- 실행일: 2026-04-18
- 감사자: codex (프롬프트 기반)

## 요약
- 총 발견 N 건 (P1: x, P2: y, P3: z)

## C1. 보안
### P1
- [파일:라인] 발견 내용 / 권장 조치
### P2
...

## C2. 엔티티-DB 스키마 정합성
...

...
```

### 3-3. codex 최종 검토
보고서가 기획서의 **범위(C1~C5)** 를 모두 다루고 있는지 codex 로 재확인.

## 4. 기능 요건 (FR)

| ID | 내용 |
|----|------|
| FR-1 | 카테고리 C1~C5 각각에 대해 **최소 1회 codex review 호출**. 응답은 **마스킹 필터(FR-5) 통과 후** 최종 보고서에 수록. **별도 raw archive 파일은 생성하지 않음** — 마스킹 전 원본이 디스크에 남지 않게 보장. |
| FR-2 | 모든 발견 사항은 **심각도 태그(P1/P2/P3)** 가 달려 있어야 하며, 태그 없는 발견은 코드 단 기준으로 임의 분류 금지 — codex 에게 태그 요구. |
| FR-3 | 보고서에 각 발견의 **파일:라인 또는 파일명 + 해당 구역** 을 포함. |
| FR-4 | 보고서 파일은 `docs/audit/{YYYY-MM-DD}-system-audit.md` 규칙. |
| FR-5 | 감사 중 codex 가 **민감한 실제 값(비밀번호, 토큰 등) 을 보고서에 그대로 복사**하지 않도록 (a) 프롬프트에 명시 — "발견은 위치·유형만, 실제 값은 `***` 로 마스킹", (b) 보고서 저장 직전 Claude 가 **민감 패턴(`password=\S+`, `sk-[A-Za-z0-9_-]+`, `Bearer\s+\S+`, `-----BEGIN .* KEY-----` 등) 을 정규식으로 재검사**하여 매치되면 자동 마스킹 후 경고 로그. (c) raw 응답이 마스킹 적용 없이 디스크에 기록되지 않도록 파이프라인 통제 (FR-1 과 일치). |
| FR-6 | Phase 2 범위(성능·의존성·테스트 등) 는 **수행하지 않음** — codex 호출 프롬프트에 명시적으로 제외. |
| FR-7 | 보고서 말미에 **사용자 검토 체크박스 영역** (각 P1 항목에 대해 "조치함/보류/불필요" 체크용) 포함. |

|----|------|
| NFR-1 | 감사 실행 시간(codex 호출 포함) ≤ 30분 — 사용자 대기 시간 관리. |

## 7. 의사결정 / 우려사항

### 7-1. 카테고리 선정 근거 — ✅ 확정
- **C1 보안** — 시스템 신뢰성 기본
- **C2 엔티티-DB** — 최근 발견된 실제 이슈(tb_contract drift) 와 동일 패턴 추가 탐지
- **C3 문서↔구현** — 기획·개발 워크플로우 정확성 기반
- **C4 Dead code** — 정리하면 유지보수 비용 절감
- **C5 민감 로그** — 운영 로그 유출 방지

### 7-2. 심각도 기준 — 확정
- **P1 (즉시)**: 실제 보안 위협 또는 이미 장애 유발 중(예: tb_contract 같은 스키마 drift 로 기능 깨짐)
- **P2 (우선)**: 잠재적 위험이나 중대 기술 부채
- **P3 (선택)**: 정리하면 좋은 수준

### 7-3. 후속 수정은 별도 스프린트 — ✅ 확정
- 이번 작업은 감사·보고서까지만
- 사용자가 보고서 보고 "무엇을 고칠지" 결정 → 각 수정은 별도 기획서

### 7-4. codex 호출 비용
- 5 카테고리 × 최소 1회 = 최소 5회 호출. 각 호출 전 prompt 를 간결하게 유지해 토큰 절약.

### 7-5. 민감 값 노출 방지 — ✅ 확정
- 프롬프트에 "발견 위치와 타입만 명시, 실제 값은 `***` 로 마스킹" 요구.
