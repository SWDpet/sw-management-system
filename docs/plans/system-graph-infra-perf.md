## 1. 배경 / 목표

### 배경
A 탭(인프라 구성도) 구현 후 실데이터로 시험한 결과, **초기 로드가 매우 느림**. 기획서 v2 NFR-1 은 "인프라 100개 기준 500ms" 였으나 실제 DB 데이터는 훨씬 큼:

| 레벨 | 수량 |
|------|------|
| 지자체 (city\|dist) | 124 |
| 시스템 | 204 |
| 서버 | 331 |
| 소프트웨어 | 983 |
| **총 노드** | **1,642** |
| **총 엣지** | **1,518** |

**부하 지점**: vis-network `barnesHut` 물리 시뮬레이션이 1600+ 노드에서 stabilize 200회 반복하며 브라우저 수 초간 블로킹.

### 목표
- **초기 렌더 ≤ 1.5 초** 로 개선 (현재는 수 초~수십 초로 추정).
- 화면에서 **SW(983 개) 는 기본 숨김** — 필요시 토글.
- **필터 UI 통일성 강화**: 인프라 유형 선택을 드롭다운 방식으로 변경 (시·도/시·군·구와 일관성).
- 구조적 변경이지만 DB/API 계약 변화 없음(Frontend 만 수정).

## 2. 사용자 시나리오

1. 관리자가 A 탭 진입 → **즉시** (1.5초 이내) 그래프 표시. 물리 시뮬레이션 없이 **계층 트리**로 배치 (상단 지자체 → 하단 서버).
2. 기본 상태에서는 **소프트웨어 노드 보이지 않음**. 필터 바에 **"소프트웨어 표시" 체크박스**가 있어 사용자가 켜면 SW 레벨이 추가 표시됨.
3. 인프라 유형 선택이 **드롭다운** 하나 (전체 / UPIS / KRAS / 기타) — 시·도/시·군·구 드롭다운과 통일.
4. 필터 변경 시 **즉시 반영** (1600+ 노드 일괄 갱신을 batch update 로 처리해 반응 지연 없음).
5. 새로고침 버튼 유지, 데이터 DB 재조회.

## 3. 화면 / UI 흐름

### 3-1. 필터 바 변경 (Before → After)

**Before (v2)**:
```
[시·도▼] [시·군·구▼] ●☑UPIS ●☑KRAS ●☑기타 [새로고침]
```

**After (v3)**:
```
[시·도▼] [시·군·구▼] [유형: 전체▼] ☐ 소프트웨어 표시 [새로고침]
```

- 유형 드롭다운: `전체 / UPIS / KRAS / 기타` 중 택일. 기본은 "전체".
- 소프트웨어 체크박스: 기본 OFF. 켜면 software 레벨 노드·엣지 표시.

### 3-2. 레이아웃 — Hierarchical

- vis-network `layout.hierarchical` 사용:
  - `direction: 'UD'` (Up-Down — 지자체 상단, SW 하단)
  - `sortMethod: 'directed'` — 엣지 방향에 따라 정렬
  - `levelSeparation: 120`, `nodeSpacing: 50` 등 (실측 튜닝)
- **Physics 끔** (`physics.enabled: false`) — 첫 렌더 시 계산 비용 없음.
- 드래그로 노드 위치 이동은 사용자가 하고 싶을 때만 활성화(선택사항).

### 3-3. 노드 모양 유지
- 지자체 `box` / 시스템 `box` / 서버 `ellipse` / 소프트웨어 `ellipse`
- 색상은 기존 INFRA_LEVEL_COLORS 유지.

### 3-4. 빈 상태 / 에러 상태
- 기존 FR-12 유지(필터로 모든 노드 숨겨지면 "필터 조건에 맞는 인프라가 없습니다").

## 4. 기능 요건 (FR)

| ID | 내용 |
|----|------|
| FR-1 | vis-network 옵션에 `layout.hierarchical` 을 설정하여 계층 트리 레이아웃 적용. `physics.enabled: false` 로 초기 시뮬레이션 제거. |
| FR-2 | 인프라 유형 필터를 **단일 select 드롭다운** 으로 변경. 옵션: `전체 / UPIS / KRAS / 기타`. 기본값 `전체`. **단일 선택 고정.** 다중 선택 승격 조건: (a) 유형 3종 이상으로 늘거나, (b) 사용자 "복수 동시 보기" 요청 누적 3건 이상 도달 시 재검토 — Phase 2 로 분리. |
| FR-3 | "소프트웨어 표시" **체크박스** 신설. 기본 체크 해제(OFF). 체크 시 software 레벨 노드 · 서버→SW 엣지 모두 표시. **페이지 새로고침·재진입 시 OFF 로 복귀** (세션 저장 안 함 — 이번 범위). |
| FR-4 | `onInfraFilter()` 의 DataSet 업데이트를 **배치 호출**(`update(arrayOfPartials)`)로 변환 — 1,642 노드에 대해 개별 update 1,642번 → 배열 1회로 축소. 엣지도 동일. |
| FR-5 | 유형 드롭다운 값이 **비어있거나 "전체"** 이면 타입 필터를 적용하지 않음 (모든 유형 통과). |
| FR-6 | "소프트웨어 표시" 해제 상태에서는, 설사 다른 필터 조건을 통과해도 software 레벨 노드와 그 엣지는 모두 숨김. SW 를 숨긴다는 이유만으로 "필터 조건에 맞는 인프라 없음" 빈 상태 메시지를 띄우지 않는다 — city/infra/server 중 하나라도 visible 이면 트리 정상 표시. 빈 상태 메시지는 모든 레벨의 **전체 노드가 0건인 경우에만** 출력. |
| FR-6-COUNT | 필터 바 또는 상세 패널 상단에 **현재 표시 중인 레벨별 카운트**를 간단히 노출 (예: `지자체 12 · 시스템 48 · 서버 77 · SW 숨김`). SW OFF 상태에서는 SW 카운트를 "숨김" 으로 표기. |
| FR-7 | 상세 패널 / 호버 툴팁 / 도메인 상속(v2 FR-2) / 민감 필드 제외(v2 FR-3) / 권한 ADMIN(v2 NFR-5) 모두 **기존 동작 유지**. 이 기획은 표현·성능만 수정. 아래 회귀 매트릭스 통과 기준 명시. |

### FR-7 회귀 매트릭스 (v2 호환성 pass 기준)

| 항목 | 검증 시나리오 | Pass 기준 |
|------|--------------|-----------|
| ADMIN 권한 | 비로그인/비-ADMIN 으로 `/api/infra` 호출 | HTTP 302 또는 403 |
| 민감 필드 배제 | `/api/infra` 응답 바디 grep | `acc_pw/sw_acc_pw/acc_id/sw_acc_id/mac_addr` + camelCase 변형 **0 hits** |
| 도메인 상속 | 서버/SW 노드에 유형 필터 적용 | 조상 시스템의 infra_type 기반으로 필터 동작 |
| 상세 패널 | 서버 클릭 | IP/OS/모델/시리얼/CPU 만 (비밀번호·MAC 없음) |
| 호버 툴팁 | 시스템 노드 호버 | 시스템명 + infra_type 표시 |
| FR-1-MERGE (C탭) | C 탭 진입 | 여전히 33 노드 정상 (A 개선이 C 에 영향 없음) |
| FR-8 | 서버 API 응답 구조(`nodes/edges/domain/attrs`) **변경 없음**. 백엔드 미수정. |

## 5. 비기능 요건 (NFR)

| ID | 내용 |
|----|------|
| NFR-1 | **초기 렌더 시간 ≤ 1.5 s** — A 탭 진입 후 노드가 화면에 배치 완료까지. (기준: 현 DB 규모 1,600+ 노드, Chrome 기준. DevTools Performance 탭으로 측정) |
| NFR-2 | 필터 조작 후 반영 시간 ≤ 300ms (배치 업데이트 덕분에). |
| NFR-3 | API 응답 시간 500ms 기준은 v2 기획서 NFR-1 **유지** — 백엔드 변경 없음. **응답 크기 기준은 재정립**: v2 의 200KB 는 "인프라 100개 기준" 이었으나 현 실데이터(시스템 204 / 서버 331 / SW 983)에서는 초과 가능. 개발계획서 단계에서 **실측 후 기준선 갱신** (예상 350~500KB). 초과 시 Phase 2 분할 로딩 검토. |
| NFR-4 | 민감 필드 제외(v2 NFR-3), 권한 ADMIN(v2 NFR-5) **유지**. |
| NFR-5 | 기능 회귀 없음 — 기존 T1~T14 중 필터 로직·shape 관련 항목은 본 개선에 맞게 갱신하되 **상세 패널/민감 필드/권한 테스트는 그대로 통과**해야 함. 구체적 Pass 기준은 FR-7 회귀 매트릭스 참조. 개발계획서 테스트 표에 매트릭스 각 항목에 대응하는 T# 반드시 포함. |

## 7. 의사결정 / 우려사항

### 7-1. Hierarchical vs 기존 force layout — ✅ 확정
- **선택**: Hierarchical
- 근거: 1,600+ 노드에서 force physics 는 초기 stabilize 가 과부하. 계층은 이미 명확하므로 tree 배치가 자연스러움.

### 7-2. SW 기본 숨김 — ✅ 확정
- 근거: SW 983개 = 전체의 60%. 대다수 사용자는 서버 레벨까지 확인이 일반적.

### 7-3. 유형 드롭다운 vs 체크박스 — ✅ 확정 (사용자 요청)
- 사용자 피드백: 체크박스 3개보다 드롭다운 1개가 UX 통일성·공간 효율 우수.
- 다중 선택 필요성은 낮음(현재 유형 UPIS/KRAS 2종 주력, 기타는 희귀).
- 복수 선택이 꼭 필요하면 차후 multi-select 컴포넌트로 확장 검토.

### 7-4. Hierarchical 노드 드래그 — 개발계획서에서 결정
- vis-network hierarchical 에서 `interaction.dragNodes` 기본값 true. 드래그 시 배치 흐트러질 수 있어 `false` 로 고정할지, 사용자 재배치 허용할지 논의.
- 권고: **드래그 허용**(기본 true). 사용자가 자유 재배치 가능하고, 새로고침하면 원래 레이아웃 복귀.

### 7-5. 기존 v2 기획서와의 관계
- 본 기획서는 v2 의 일부 FR(레이아웃·필터 UI) 를 대체하지만, v2 전체를 폐기하지 않음. 구조·권한·민감정보 등 핵심 요구사항은 v2 그대로.

## 9. 리스크 평가

| 리스크 | 수준 | 완화책 |
|--------|------|--------|
| Hierarchical 레이아웃이 특정 데이터 형태에서 겹침/뭉침 | 중간 | `nodeSpacing`/`levelSeparation` 실측 튜닝, 개발계획서에 기본값 명시 |
| SW 기본 숨김을 모르는 사용자가 "SW 안 보임" 으로 착각 | 낮음 | 체크박스 라벨 `소프트웨어 표시`, 빈 상태 아님에도 부재감 주지 않도록 UI 명확히 |
| 드롭다운 단일 선택으로 UPIS+KRAS 동시 보기 희귀 수요 불충족 | 낮음 | 기본 "전체" 옵션 제공. 필요시 별도 Phase 에서 multi-select |
| 배치 update 로직에 버그가 있으면 DataSet 불일치 → 일부 노드 잘못 표시 | 중간 | 개발계획서에서 회귀 테스트 T# 강화 (필터 조합 커버리지) |
| hierarchical 에서 dragNodes 허용 시 사용자가 실수로 흐트러뜨림 | 낮음 | 새로고침 시 재배치로 복구 가능 (사용자 안내) |
