🔒 전체 HTTPS 전환 — 실행 가이드

SW Manager 운영(Windows + 외부 Tomcat 11 + ROOT.war) · nginx TLS 종단 + win-acme(Let's Encrypt)
근거 런북: docs/exec-plans/https-reverse-proxy-deploy.md · 본인 도메인(가비아) 확정본

확정 도메인 swbms.geonuris.net 진행 0/7
목표 최종 형태 (포트 없이 표준 443): [외부 갤탭/PC] ──https://swbms.geonuris.net──▶ [iptime 공유기 공인IP 443] ──포트포워딩──▶ [Windows 서버 443 nginx TLS종단] ──http──▶ [Tomcat 127.0.0.1:8093 (ROOT.war)]
각 단계 헤더를 클릭하면 펼쳐지고, 체크박스로 완료 표시(이 브라우저에 저장)됩니다. 순서대로 진행하세요 — 특히 Step 1(공유기 포트 비우기) → Step 4(80 먼저) → Step 5(발급) → Step 6(443) 순서가 중요합니다.
1

DNS A레코드 확인 + iptime 80/443 점유 해소 ★현재 막힌 지점

직접 — 가비아 DNS + iptime 공유기
원인: 포트를 안 붙이면(=80/443) iptime 공유기가 그 포트를 자기 관리자 페이지로 점유해서, 서버가 아니라 공유기 화면이 떴습니다. 그래서 8092로 우회 중이었던 것. DNS/SRV 문제가 아니라 공유기 포트 문제입니다.
① 가비아 DNS — A레코드 확인 (SRV 아님):
  • 가비아 → 도메인 → DNS 관리 → 레코드 타입 A, 호스트 swbms, 값 = 운영 공인 IP
  • ⚠ SRV 레코드는 입력하지 마세요 — 브라우저는 SRV를 조회하지 않아 웹 접속엔 무효입니다. swbms.geonuris.net:8092 가 이미 붙으면 A레코드는 이미 정상입니다(확인만).
② iptime 공유기 — 80/443 원격관리 포트 비우기:
  • iptime 관리자(192.168.0.1) → 고급 설정 → 보안 기능 → 공유기 접속/관리 포트 설정 (펌웨어에 따라 "원격 관리 포트")
  • HTTP 관리 포트 80 → 다른 값(예 8090)으로 변경하거나 외부 원격관리 해제
  • HTTPS 관리 포트가 443이면 그것도 다른 값으로 변경
  • ⚠ 변경 후 공유기 관리자 접속도 새 포트로 (192.168.0.1 내부 접속은 그대로)
메뉴명은 iptime 펌웨어 버전마다 조금씩 다릅니다("보안 기능" 또는 "시스템 관리" 아래). "원격 관리 포트" / "공유기 접속 포트" 항목을 찾으세요.
2

포트포워딩 80/443 → 서버 + 8093·8092 외부차단

직접 — 공유기 + Windows 방화벽
  • iptime 포트포워딩: 외부 80 → 서버 80, 외부 443 → 서버 443 (80은 win-acme http-01 챌린지에 필요)
  • 지금 쓰던 8092 포워딩은 표준 443 완성 후 제거 (그 전까진 둬도 됨)
  • Tomcat 8093 외부차단 — 둘 중 하나:
① Windows Defender 방화벽 인바운드 규칙으로 8093 외부 차단 (관리자 PowerShell):
New-NetFirewallRule -DisplayName "Block-Tomcat-8093-External" -Direction Inbound -LocalPort 8093 -Protocol TCP -Action Block -RemoteAddress Internet
② 또는 Tomcat conf/server.xml 의 8093 Connector 에 address="127.0.0.1" 추가 (로컬 바인드 제한).
차단 방식을 기억해두세요 — 롤백 절차가 방식별로 다릅니다(맨 아래).
3

Windows nginx 설치

직접 — 설치
  • nginx for Windows 를 C:\nginx\ 에 설치
  • ACME 챌린지용 webroot + 인증서 폴더 생성
  • 서비스 등록(nssm/winsw)으로 부팅 자동시작 권장
New-Item -ItemType Directory -Force C:\nginx\html\.well-known\acme-challenge | Out-Null
New-Item -ItemType Directory -Force C:\nginx\ssl | Out-Null
New-Item -ItemType Directory -Force C:\nginx\ssl-staging | Out-Null
서비스 등록 예 (nssm 사용 시):
nssm install nginx C:\nginx\nginx.exe
nssm set nginx AppDirectory C:\nginx
4

nginx 80 챌린지 블록 먼저 적용

제가 작성 — 발급 전 80 응답 필수
순서 중요: 443 인증서가 아직 없으므로 80 블록만 먼저 올립니다. http-01 챌린지가 80에서 응답해야 발급됩니다.
C:\nginx\conf\nginx.conf 전체 (이 상태로 1차 기동):
worker_processes  auto;

events {
    worker_connections  1024;
}

http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile        on;
    keepalive_timeout  65;

    # === 80: ACME 챌린지 + HTTPS 리다이렉트 ===
    server {
        listen 80;
        server_name swbms.geonuris.net;
        location /.well-known/acme-challenge/ { root C:/nginx/html; }   # 챌린지 (redirect보다 우선)
        location / { return 301 https://$host$request_uri; }            # http -> https
    }
}
설정 검증 후 기동:
cd C:\nginx
.\nginx.exe -t
.\nginx.exe              # 최초 기동 (또는  nssm start nginx)
  • 외부망(휴대폰 등)에서 http://swbms.geonuris.net/.well-known/acme-challenge/ping 접속 → nginx 응답(404라도 OK) 확인. 여기서 공유기 페이지가 뜨면 Step 1·2 가 덜 된 것.
5

win-acme 로 인증서 발급

직접 — wacs.exe 실행
① 먼저 staging 리허설 (운영 rate-limit 무소모, 전용 경로):
C:\tools\win-acme\wacs.exe --test --notaskscheduler `
  --source manual --host swbms.geonuris.net `
  --validation filesystem --webroot C:\nginx\html `
  --store pemfiles --pemfilespath C:\nginx\ssl-staging
리허설 성공하면 staging 폴더 삭제: Remove-Item -Recurse -Force C:\nginx\ssl-staging · staging 인증서는 신뢰되지 않으니 절대 운영 경로에 두지 마세요.
② 실제 운영 발급(production):
C:\tools\win-acme\wacs.exe `
  --source manual --host swbms.geonuris.net `
  --validation filesystem --webroot C:\nginx\html `
  --store pemfiles --pemfilespath C:\nginx\ssl `
  --installation none
③ 발급된 실제 PEM 파일명 확인 (Step 6 경로에 반영):
Get-ChildItem C:\nginx\ssl
  • 보통 swbms.geonuris.net-chain.pem(fullchain) + swbms.geonuris.net-key.pem(키) 생성
  • win-acme 가 자동갱신 예약작업(Task Scheduler) 생성 — post-renewal 로 nginx -s reload 등록 권장
6

nginx 443 TLS 블록 추가 + reload

제가 작성
Step 4 의 http { } 안, 80 server 블록 아래에 추가 (PEM 파일명은 Step 5 실제 값으로):
    # === 443: TLS 종단 → Tomcat ===
    server {
        listen 443 ssl;
        server_name swbms.geonuris.net;

        ssl_certificate     C:/nginx/ssl/swbms.geonuris.net-chain.pem;   # 실제 파일명으로
        ssl_certificate_key C:/nginx/ssl/swbms.geonuris.net-key.pem;     # 실제 파일명으로
        ssl_protocols       TLSv1.2 TLSv1.3;

        client_max_body_size 50m;   # 기본 1MB -> 업로드 413 방지 (앱 한도에 맞춰 조정)

        # HSTS 초도엔 짧게. 안정화 후 31536000 상향
        # ⚠ includeSubDomains/preload 금지 — geonuris.net 전 서브도메인에 강제됨
        add_header Strict-Transport-Security "max-age=300" always;

        location / {
            proxy_pass         http://127.0.0.1:8093;
            proxy_http_version 1.1;
            proxy_read_timeout 300s;
            proxy_set_header   Host              $host;
            proxy_set_header   X-Real-IP         $remote_addr;
            proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header   X-Forwarded-Proto https;
            proxy_set_header   X-Forwarded-Host  $host;
            proxy_set_header   X-Forwarded-Port  443;
        }
    }
cd C:\nginx
.\nginx.exe -t
.\nginx.exe -s reload
7

앱 설정 1줄 + ROOT.war 재배포

제가 편집 가능
.properties 는 값 뒤 주석 금지 — 주석은 반드시 별도 줄로. (인라인 주석 쓰면 값이 통째로 잘못 파싱됨)
application-prod.properties 에 추가:
# 프록시 뒤 X-Forwarded-* 인식 (없으면 secure쿠키/리다이렉트/isSecure 오동작)
server.forward-headers-strategy=framework
# server.servlet.session.cookie.secure=true  (이미 설정됨)
  • ROOT.war 재배포 (배포 절차는 기존 운영 배포 메모 그대로)
이 한 줄 편집·재배포는 제가 해드릴 수 있습니다 — 말씀만 주세요.

✅ 검증 체크리스트 (프록시 443 경유로)

↩ 롤백 (문제 시)

  • 무중단: nginx 서비스 중지 → Tomcat(8093) 그대로. 단 8093 외부차단 했으면 아래 방식별 복귀 필요
  • 방화벽으로 차단한 경우 → Defender 규칙 비활성화 + iptime 포트포워딩(외부 8092→서버 8093) 임시 복구
  • Connector address=127.0.0.1 로 차단한 경우 → address 제거 + Tomcat 재기동 + 포트포워딩 복구
  • properties 동반 롤백: HTTP 로 완전 복귀 시 forward-headers-strategy 제거 + cookie.secure=false 로 재배포
  • iptime 원격관리 포트를 80/443 으로 되돌리면 다시 공유기 페이지가 점유 — 롤백 시 그대로 둘지 판단
⚠ 본인 도메인(swbms.geonuris.net)이라 *.iptime.org 공유 rate-limit 위험은 해소됨. 다만 TLS 본격 하드닝(cipher/OCSP)·rate-limit·접근제한은 별도 보안 sprint 권장. HSTS includeSubDomains/preload 는 geonuris.net 전체에 영향 주므로 신중히.
