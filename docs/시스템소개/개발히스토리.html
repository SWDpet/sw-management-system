01. 이 시스템이 무엇인가요?

개발을 전혀 모르셔도 괜찮습니다. 먼저 "무엇을 만들었는지"부터 일상 언어로 설명합니다.

SW 사업관리 시스템은 우리 부서가 진행하는 여러 소프트웨어(SW) 사업을 한곳에서 관리하는 사내 웹사이트입니다. 지금까지 엑셀과 한글 문서, 폴더 여기저기에 흩어져 있던 일들을 — 사업 정보, 담당자, 서버 장비, 계약·준공 서류, 점검 결과까지 — 하나의 화면에서 보고 관리할 수 있게 모은 것입니다.

비유하자면 — 회사의 모든 사업을 한눈에 보는 "관제탑"입니다. 어떤 지자체의 어떤 시스템을, 누가, 언제까지, 얼마에 진행 중인지, 서류는 어디까지 됐는지를 관제탑 화면 하나로 내려다보는 셈이죠.

주로 이런 일을 합니다

📊 대시보드
진행 중인 SW 사업의 현황·금액·단계를 한 화면에 요약해 보여줍니다.
📁 사업 관리
사업을 등록·수정하고 지역·시스템·연도로 검색합니다.
📄 문서 자동 작성
착수계·기성계·준공계 같은 공식 한글(HWP) 서류를 버튼 한 번으로 만들어 줍니다.
🔍 점검내역서
현장 서버를 점검하고 그 결과를 표준 양식의 PDF 점검표로 만들어 줍니다.
💰 견적서
인건비·요율을 자동 계산해 견적서를 작성하고 대장으로 관리합니다.
🛠 장애처리·업무지원
고객 요청과 장애 대응 내역을 기록하고, 비슷한 사례의 해결책을 추천합니다.
🖥 인프라(서버) 관리
각 사업의 서버·장비 정보를 자산처럼 관리합니다.
👤 사용자·권한·로그
직원 계정 승인, 메뉴별 권한, 모든 작업 기록(로그)을 관리합니다.
💡 "커밋(commit)"이라는 말이 자주 나옵니다. 커밋은 작업을 한 번 저장하면서 "무엇을 바꿨는지" 한 줄로 적어 두는, 일종의 작업 일지 한 칸입니다. 이 문서의 숫자(335건)는 그렇게 쌓인 작업 일지의 칸 수입니다. 이 기록을 따라가면 시스템이 어떻게 자라났는지 그대로 보입니다.

03. 시기별 상세 이야기 — ①·②단계

이제 각 단계에서 실제로 무슨 일이 있었는지, 조금 더 들어가 봅니다.

① 4월 초·중순 — "쓸 수 있는 기능을 빠르게"

이 저장소의 첫 기록은 2026-04-03입니다("이전 작업을 옮겨오기 전 첫 저장"). 즉, 시스템 자체는 그 전부터 있었고, 이 시점에 본격적인 기록 관리가 시작된 셈입니다.

  • 문서·견적·점검 대규모 개선(4/13): 공식 서류(한글 HWPX) 자동 생성, 점검내역서의 상세 화면·미리보기·PDF 출력을 한꺼번에 키웠습니다.
  • 태블릿 대응(4/13): 현장에서 쓰는 갤럭시탭 세로 화면에서도 7개 점검 화면이 잘 보이도록 맞췄습니다(이른바 '반응형').
  • 팀 진행율 모니터링(4/14): 팀원들의 진행 상황을 한 화면에서 보는 대시보드를 추가했습니다.
  • 협업 방식 문서화(4/14): 'codex'라는 외부 점검 도구로 코드를 한 번 더 검증하는 작업 흐름을 정해 글로 남겼습니다.
비유 — 집을 짓는다면 이 시기는 방을 빠르게 늘리는 단계입니다. 일단 살 수 있게 공간을 만든 거죠.

② 4월 중·하순 — "점검하고 기초를 다지다"

  • 보안 감사(4/19): 외부 도구로 시스템을 훑어 문제점을 1~3순위로 분류하고, 비밀번호 노출·권한 검사 누락·민감정보 같은 보안 위험을 차례로 고쳤습니다.
  • 데이터 구조 재정리(4/20~4/22): 오래되어 안 쓰는 데이터 표를 정리하고, 중복을 없애고, 항목들을 표준화했습니다(개발 용어로 '정규화'·'리팩토링'). 화면은 그대로지만 속이 깔끔해진 작업입니다.
  • 로그 데이터 정제(4/20): 잘못 기록된 작업 로그 27건을 바로잡고, 앞으로 오염되지 않도록 안전장치를 넣었습니다.
  • 개발 문서 전면 재편(4/24): 흩어진 설명 문서들을 진입점 하나로 모으고 폴더를 재배치했습니다(P0~P7 단계).
비유 — 늘린 방들을 두고 배관·전기·단열을 점검하고 정비하는 단계입니다. 겉보기엔 비슷해도 안전하고 오래 쓸 수 있게 됩니다.
💡 'codex'란? 사람이 짠 코드를 한 번 더 들여다보는 인공지능 검토 도우미입니다. 이 프로젝트는 "사람이 만들고 → AI가 검토하고 → 사람이 최종 승인"하는 방식을 거의 모든 작업에 적용했습니다(8장 참고).

04. 시기별 상세 이야기 — ③·④단계

③ 5월 — "현장 점검 자동화" 대장정 + "디자인 통일"

5월은 가장 길고 치열했던 달입니다(작업 기록 138건). 두 갈래의 큰 작업이 동시에 진행됐습니다.

가. 현장 점검 자동화 프로그램

지자체 현장 서버를 직접 점검하던 일을, 현장에 설치하는 작은 자동 점검 프로그램이 대신하도록 만들었습니다.

  • QR 반출(5/11~): 인터넷이 안 되는 폐쇄망 현장에서도, 점검 결과를 QR코드로 찍어 안전하게 가지고 나오는 방식.
  • 원격 점검(5/16~5/17): 비밀번호를 안전하게 보관(DPAPI)하고, 원격 접속(Telnet·SSH)으로 서버 상태를 수집. 오래된 윈도우 서버(2012)에서도 한글이 안 깨지도록 세심하게 맞췄습니다.
  • 점검표 자동 생성(5/13~5/27): 수집한 값으로 표준 점검내역서를 자동으로 채우고(시안 D v5·v6), 그래프까지 그려 넣었습니다.
  • 점검 범위 분기(5/30~5/31): 유지보수 유형에 따라 점검 항목을 자동으로 달리 보여주도록 했습니다.

나. 디자인 전면 통일 (5/31, 하루 52건!)

그동안 화면마다 조금씩 달랐던 색·버튼·표 모양을, 하나의 색 기준표(디자인 토큰)로 통일했습니다. 견적서 작성 화면(3,209줄), 착수계, 점검내역서 작성 등 40개가 넘는 화면을 하루 만에 일괄 정비한 기록이 남아 있습니다.

비유 — 가게 인테리어를 브랜드 색으로 통일한 것입니다. 어느 화면에 가도 같은 회사가 만든 느낌이 들도록.

④ 6월 — "실무에 맞춰 다듬고 넓히다"

  • 서류 편의 기능(6/9~6/10): 최종 날인(도장)본 스캔 PDF 보관, 공문 서식 보정, 검색 결과 서류 일괄 ZIP 다운로드.
  • 화면 구조 개편(6/11): 상단 메뉴를 왼쪽 사이드바로 바꾸고 모바일까지 대응, 대시보드를 메인으로 승격했습니다.
  • 업무계획 연결(6/11): 시·도 → 시·군·구 → 시스템으로 이어 선택하는(캐스케이드) 방식 도입.
  • 장애처리·업무지원 개편(6/13~6/17): 조직·인원(직원 85명 등록), 담당 엔지니어와 요청자(공무원/업체), 그리고 지식베이스 — 증상을 넣으면 비슷한 사례의 원인·조치를 추천하는 기능을 더했습니다.
  • 로그·운영문서 정리(6/16~6/17): 로그 관리 화면을 개선하고 통계를 넣었으며, 운영문서의 상세·수정·삭제 흐름을 정돈했습니다.
비유 — 집들이 전에 가구 배치를 바꾸고 살림 도구를 채워 넣는 단계입니다. 실제로 살아보며 불편을 없앤 거죠.

07. 지금까지 완성된 것

3개월의 작업으로 현재 시스템에는 다음과 같은 기능들이 들어 있습니다.

통합 대시보드
시스템별 사업 현황·금액·완료/진행 증감을 메인 화면에서 한눈에.
사업 관리
탭으로 나뉜 등록/수정 폼, 지역·시스템·연도 검색, 칸반식 단계 표시.
공식 문서 자동 생성
착수계·기성계·준공계 한글 공문/본문 자동 작성, 날인본 스캔 보관, 일괄 ZIP 다운로드.
점검내역서 + 자동 점검
현장 자동 점검 프로그램(QR·원격), 수집값 자동 채움, HTML→PDF 표준 점검표.
견적서·견적 대장
인건비·요율·낙찰율 자동 계산, 절사 단위 적용, 미리보기·대장 관리.
장애처리·업무지원
담당 엔지니어·요청자(공무원/업체) 관리, 시간 기록, 운영문서 상세·수정·삭제.
지식베이스(KB) 추천
증상→원인→조치 사례를 등록·승인·축적하고, 비슷한 사례를 자동 추천.
인프라(서버) 자산
서버·장비 정보 등록/상세, 점검 시 현장값과 차이 알림.
사용자·권한·로그
회원가입 승인, 메뉴별 권한(없음/조회/편집), 모든 작업 로그 기록·통계.
디자인 시스템
중앙 색 기준표(토큰)로 40여 화면 통일, 전사 사이드바 + 모바일 반응형.
조직·인원 관리
전 직원 조직도(85명) 시드, 본부-팀-인원 연결.
운영 보안(HTTPS)
리버스 프록시 기반 HTTPS 전환 런북, 외부 노출 차단 방화벽 형상관리.

내부적으로는 화면 70여 개, 백엔드 처리 모듈(컨트롤러) 20여 개 규모로 구성되어 있습니다.

08. 어떻게 개발했나 — 일하는 방식

이 프로젝트의 특별한 점은 '혼자 빨리'가 아니라 '검토를 거쳐 안전하게'였다는 것입니다.

① 사람 + AI 이중 검토 방식

거의 모든 작업은 다음 흐름을 거쳤습니다. 새 기능을 바로 만들지 않고, 먼저 계획서를 쓰고 → AI(codex)가 검토하고 → 사람이 최종 승인한 뒤 → 만들고 → 다시 검증했습니다.

요청 → 기획서 작성 → (화면 바뀌면) 디자인·데이터 검토 → AI 검토 → 사람 승인 → 개발계획 → AI 검토 → 사람 승인 → 구현 → AI 검증 → "작업 완료" 저장

기록에 'codex 검토 반영', 'v0.2 → v3 (codex 2차 반영)' 같은 표현이 자주 보이는 이유입니다. 한 기능을 위해 계획서를 여러 번 고쳐 쓴 흔적이죠.

② '가상 팀' 체계

실제로는 한 사람이 진행했지만, 기획·DB·디자인·개발·테스트 다섯 역할을 나눠 스스로 점검하는 체계를 갖췄습니다. 화면이 바뀌는 작업에는 '디자인팀 자문'을 반드시 거치도록 규칙을 정했습니다.

③ 신중하게 멈춘 결정들도 기록

모든 시도가 성공한 것은 아닙니다. 계약서 PDF 자동 파싱 기능(5/4)은 만들었다가 사용자 판단으로 취소했고, 일부 보안 작업은 의도적으로 보류했습니다. 이런 "안 하기로 한 결정"까지 남긴 점이 이 프로젝트 기록의 강점입니다.

💡 한마디로, "빠르게 만들기"보다 "틀리지 않게 만들기"에 무게를 둔 개발이었습니다. 그래서 화면 수에 비해 보안 감사·데이터 정리·문서화에 들인 시간이 큽니다.

09. 용어 사전 & 맺음말

이 문서에 나온 개발 용어를 한 줄씩 풀었습니다.

커밋(commit)
작업을 한 번 저장하며 "무엇을 바꿨는지" 한 줄로 적어 두는 작업 일지 한 칸.
저장소(repository)
그 작업 일지와 모든 코드가 쌓이는 보관함. 이 시스템의 보관함에는 335칸이 쌓였습니다.
codex
사람이 짠 코드를 한 번 더 검토하는 AI 검토 도우미.
리팩토링(refactoring)
겉보기 기능은 그대로 두고 속 구조만 깔끔하게 정리하는 일. '대청소'에 가깝습니다.
정규화(normalization)
데이터를 중복 없이 표준 형태로 정리하는 것. 같은 정보가 여기저기 흩어지지 않게.
반응형(responsive)
PC·태블릿·휴대폰 등 화면 크기에 맞춰 레이아웃이 자동으로 바뀌는 것.
디자인 토큰
색·간격·글꼴 같은 디자인 값을 한 곳에 정해 둔 '기준표'. 모든 화면이 같은 기준을 씁니다.
대시보드(dashboard)
여러 정보를 한 화면에 요약해 보여주는 종합 상황판.
캐스케이드(cascade)
앞 선택에 따라 다음 선택지가 좁혀지는 방식. 예: 시·도를 고르면 그 안의 시·군·구만 보임.
지식베이스(KB)
'증상→원인→조치' 사례를 쌓아 두고, 비슷한 일이 생기면 꺼내 보는 해결책 모음.
HTTPS / 리버스 프록시
웹 접속을 암호화해 안전하게 만드는 방식과, 그 앞단에서 교통정리를 하는 장치.
핫리로드(hot reload)
프로그램을 껐다 켜지 않아도 바뀐 코드가 바로 반영되는 기능.
