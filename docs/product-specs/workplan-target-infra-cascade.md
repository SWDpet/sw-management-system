## 0. 한 줄 요약

`업무계획 등록 > 기본정보 > 대상 인프라`의 **단일 드롭다운**을 **시도 → 시군구 → 시스템 3단 캐스케이드**로 바꾸고, **계약(등록)되지 않은 곳에 가는 업무지원**도 대상으로 지정할 수 있도록 `tb_work_plan`에 지역·시스템 컬럼을 추가한다.

---

## 1. 배경 / 목적

- 현재 대상 인프라는 `tb_infra_master` 전체를 `"시도 시군구 - 시스템"` 한 줄로 나열한 **단일 `<select>`** 하나다(`workplan-form.html` 184~193행). 인프라가 늘면 길어져 찾기 어렵다.
- 더 큰 문제: 대상은 **등록된 Infra 만** 고를 수 있다. 그런데 **업무지원(SUPPORT)** 은 아직 **계약/등록되지 않은 기관**에 가는 경우가 있고, 지금은 이를 업무계획의 "대상"으로 지정할 방법이 없다(`infra_id` 가 없으면 무대상으로만 저장).
- 시도/시군구는 이미 전국 행정구역 마스터(`sigungu_code`, 279건)가 있어, **미계약 지역도 코드 기준으로 선택**할 수 있다.

## 2. 범위 (In / Out)

**In**
- `workplan-form.html` 의 대상 인프라 입력을 **3단 캐스케이드**(시도→시군구→시스템)로 교체.
- 시군구의 **등록 인프라 목록** + **"목록에 없음(직접입력)"** 분기.
- `tb_work_plan` 에 **미계약 대상용 additive 컬럼**(지역코드·시도·시군구·시스템명) 추가.
- 등록/수정 화면 진입 시 기존 저장값으로 **캐스케이드 초기 복원**.
- 저장(`/workplan/save`)·DTO·서비스의 대상 처리 확장.

**Out**
- 성과통계 화면 자체의 신규 집계 리포트 추가는 본 건 Out(데이터는 집계 가능하게 저장하되, 신규 통계 화면은 후속).
- Infra 마스터 관리(서버관리) 화면 변경 없음.
- 캘린더/프로세스 현황의 표시 로직 대규모 개편 없음(대상 라벨 표기만 정합).
- 점검주기/점검현황(삭제된 기능, `8cbd54c` 보존) 복원은 무관·별건.

## 3. 사용자 스토리

> ① 담당자로서, 업무계획을 등록할 때 대상지를 **시도 → 시군구 → 시스템** 순으로 좁혀가며 빠르게 고르고 싶다.
> ② 담당자로서, **아직 계약 안 된 기관에 업무지원**을 가야 할 때, 그 **지역과 시스템(직접입력)** 을 대상으로 지정해 업무계획을 남기고 싶다.

## 4. 기능 요구사항 (FR)

| ID | 요구사항 |
|---|---|
| FR-1 | 대상 인프라 입력을 **시도 / 시군구 / 시스템 3단 종속 드롭다운**으로 교체. 상위 미선택 시 하위 비활성. |
| FR-2 | **시도·시군구 소스 = `sigungu_code`**(행정구역 마스터). 시도=`sido_name` distinct, 시군구=선택 시도의 `sgg_name`(값은 `adm_sect_c`). 전국 노출(미계약 지역 포함). |
| FR-3 | **시스템 단계** = 선택 시군구에 해당하는 **등록 인프라(`tb_infra_master`) 목록**(값=`infra_id`, 라벨=`sys_nm`) + 맨 끝 **"목록에 없음 / 직접입력"** 옵션. |
| FR-4 | 등록 인프라 선택 시 기존대로 **`infra_id` 저장**(현 동작 유지, 회귀 없음). 이때 지역 보조필드는 인프라값으로 채움(FR-9). |
| FR-5 | **"목록에 없음"** 선택 시 시스템명 **텍스트 입력**을 노출하고, 저장 시 `infra_id=NULL` + `region_code`(시군구코드)·`region_city_nm`·`region_dist_nm`·`target_sys_nm` 저장. |
| FR-6 | **(확정 A)** 미계약 직접입력("목록에 없음")은 **업무유형 = 업무지원(SUPPORT) 일 때만** 노출·허용. 그 외 유형은 **등록 인프라 선택 필수**. planType 변경 시 직접입력 칸 토글. |
| FR-7 | **수정 화면 복원**: 기존 `infra_id` 가 있으면 그 인프라의 시도/시군구/시스템으로, 없고 `region_code` 가 있으면 그 지역+직접입력값으로 캐스케이드 초기 선택을 복원. |
| FR-8 | **비동기 캐스케이드 API**: 시군구 목록(`?sido=`), 시스템 목록(`?admSectC=`) 을 서버에서 JSON 으로 제공. 인증·권한 필요. |
| FR-9 | **저장 정합성**: `infra_id` 와 미계약 입력이 동시에 오면 **`infra_id` 우선**(미계약 필드 무시). 어느 쪽도 없으면 "대상 없음"(현행 "선택 안함" 허용 유지). |
| FR-9b | **(확정 채움)** **대상이 있을 때** region 4필드를 채운다. **계약 대상**: 서버가 `infra_id` 로 **Infra 를 재조회**해 `city_nm/dist_nm/sys_nm` + 시군구코드를 채운다(클라이언트 hidden 값 불신 — 서버 재계산 원칙). **미계약**: 직접입력·선택값. → 성과통계를 region 컬럼 단일 출처로 집계. **예외**: "대상 없음"(아래 FR-9) 저장은 region 4필드 **전부 null** 허용. |
| FR-9c (보안 게이트) | 서버측: ① `planType != SUPPORT` 인데 `infra_id` 없이 미계약 필드가 오면 **거부**(FR-6 우회 차단). ② `infra_id` 도 미계약 필드도 없으면 **"대상 없음"으로 정상 저장**(planType 무관, region null). |
| FR-10 (회귀방지·중요) | **표시 정합**: `WorkPlanDTO.fromEntity` 와 캘린더 `infraName` 빌더가 현재 **`infra` 있을 때만** `시군구-시스템` 라벨을 만든다 → **미계약(infra_id=null)일 때 region 4필드로 fallback** 하도록 보강. 목록/캘린더/상세 모두 `시군구 - 시스템` 동일 포맷. (미보강 시 미계약 대상이 빈칸으로 보이는 회귀.) |
| FR-11 (검증) | 서버측 검증: `region_code` 는 `sigungu_code.adm_sect_c` **존재값만** 허용(미존재 거부). `target_sys_nm` = **trim 후 1~100자**, 제어문자(`\x00-\x1F`)·HTML 특수문자 제거/이스케이프, 빈문자면 미계약 저장 거부. |
| FR-12 (광역/도 단위) | 시도 선택 시 시군구 목록의 **self-행**(`sgg_name=sido_name`, 예 울산 `31000`·경남 `64800`)을 **"(본청/도청)"** 라벨로 노출 → 도/광역 단위 자체를 대상으로 지정 가능(§5-4). **매핑 책임**: Infra 의 `도청`/`본청` 표기 → self-행 변환은 **서비스 계층(WorkPlanService 의 지역해석 메서드)** 에서 수행(컨트롤러/뷰 아님). |

## 5. 데이터 / 스키마 설계

### 5-1. 참조 테이블
- `sigungu_code` : `adm_sect_c`(PK, 5자리 행정코드), `sido_name`(시도), `sgg_name`(시군구), `full_name`, `instt_c`.
- `tb_infra_master`(Infra) : `infra_id`, `city_nm`, `dist_nm`, `sys_nm`, `dist_cd`, `org_cd`.
- `tb_work_plan`(WorkPlan) : `infra_id`(FK nullable), `plan_type`, …

### 5-2. `tb_work_plan` additive 컬럼 (nullable, 멱등 ALTER)
| 컬럼 | 타입 | 의미 |
|---|---|---|
| `region_code` | VARCHAR(10) | 시군구코드(`sigungu_code.adm_sect_c`). **모든 저장 시 채움**(FR-9b): 계약=선택 시군구 코드, 미계약=직접선택 코드. |
| `region_city_nm` | VARCHAR(40) | 시도명(표시·통계 denormalize). 계약=Infra.city_nm, 미계약=선택 시도. |
| `region_dist_nm` | VARCHAR(40) | 시군구명(표시·통계 denormalize). 계약=Infra.dist_nm, 미계약=선택 시군구. |
| `target_sys_nm` | VARCHAR(100) | 시스템명. 계약=Infra.sys_nm 복사, 미계약=직접입력값. |

> **채움 정책(FR-9b 확정)**: 계약·미계약 모두 region 4필드를 채워 성과통계를 단일 출처로 집계. denormalize 이나 업무계획은 *그 시점 기록*이라 Infra 후속 변경과 무관(허용).

> 모두 nullable·additive → 기존 행/코드 무영향. `Document.region_code` 선례와 동일 철학(사업/인프라 독립 대상 식별).

### 5-3. 시군구 ↔ 인프라 매칭 키 (✅ 운영DB 실측 2026-06-11 → 이름 매칭 확정)
운영DB `tb_infra_master`(205건) 실측:
- **코드 매칭** `Infra.dist_cd == sigungu_code.adm_sect_c` → **28건(14%)** 만 매칭(`dist_cd` 채움 41건뿐) → **사용 불가**.
- **이름 매칭** `Infra.city_nm/dist_nm == sigungu_code.sido_name/sgg_name` → **197건(96%)** 매칭 → **이게 1차 키**.
- 결론: 시스템 목록 조회는 **이름 매칭(시도명+시군구명) 우선**. API 는 `admSectC` 를 받아 `sigungu_code` 에서 `sido_name/sgg_name` 으로 변환 후 Infra 이름으로 조회.
- 미매칭 잔여 **8건(205−197)** 실측(2026-06-11) → **구제시 문제 아님**. (구제시 "고양시 덕양구" 등은 Infra 가 구 단위로 저장돼 이미 이름매칭됨.)

### 5-4. 시도(광역/도) 단위 대상 — "도청/본청" (✅ 실측으로 해결)
`sigungu_code` 에 **17개 시도 전부 self-행(`sgg_name = sido_name`)** 이 존재한다 → 도/광역 단위 자체를 대상으로 지정 가능:
| 시도 | self-행 `adm_sect_c` |
|---|---|
| 울산광역시(본청) | `31000` |
| 서울 11000 · 부산 26000 · 대구 27000 · 인천 28000 · 광주 29000 · 대전 30000 · 세종 36110 · 제주 50000 | (광역시 self) |
| 경기 64100 · 충북 64300 · 충남 64400 · 전남 64600 · 경북 64700 · 경남 64800 · 강원 65300 · 전북 65400 | (도 self) |

**8건 매핑 규칙(확정)**:
| Infra 표기 | 매핑 |
|---|---|
| `도청`(경남/경북/충남/충북) | 그 시도 self-행 → 경남 `64800`, 경북 `64700`, 충남 `64400`, 충북 `64300` |
| `본청`(울산광역시 등) | 그 광역시 self-행 → 울산 `31000` |
| `고성군(강원도)` / `고성군(경남)` | 괄호 접미사 제거 후 **시도+시군구**로 유일 매핑(강원 51820 / 경남 해당코드) |
| `당진시(가상화)` | 접미사 제거 → 충남 당진시 |
| `군위군` | 행정개편(2023 경북→대구) → 현행 **대구광역시 군위군** 코드(alias) |

→ 캐스케이드는 시도 선택 시 시군구 목록(`WHERE sido_name=X`)에 **self-행이 자연 포함**되므로, UI 에서 그 항목 라벨만 **"(본청/도청)"** 으로 표기하면 도/광역 단위 대상이 그대로 선택된다(별도 특수처리 불필요).

## 6. UI 설계 (🎨 디자인팀 자문 대상)

### 6-1. 레이아웃 (기본정보 "대상 인프라" 행 교체)
```
대상 인프라   [ 시도 ▾ ] [ 시군구 ▾ ] [ 시스템 ▾ ]
                                   └ "목록에 없음" 선택 시 → [ 시스템명 직접입력 ____ ]
```
- 3개 `<select>` 가로 배치(반응형 시 wrap), 상위 미선택 시 하위 `disabled`.
- **"목록에 없음" 옵션은 업무유형=업무지원(SUPPORT)일 때만** 시스템 select 에 노출(FR-6). 선택 시 그 아래 `text input`(시스템명 직접입력) 표시(JS 토글). 업무유형을 SUPPORT 외로 바꾸면 직접입력 칸·옵션 숨김.

### 6-2. 🎨 디자인팀 가상 자문 결과
- **디자인 토큰만 사용**(색 하드코딩 금지): `--border2`/`--primary`/`--surface-2` 등 기존 폼 토큰 재사용 → **다크모드 자동 일관**.
- 신규 색/`:root` self-reference 추가 **없음**(기존 `.form-table select` 스타일 그대로 상속) → specificity 충돌 없음.
- 접근성: 각 select 에 `<label>`/`aria-label`, 종속 로딩 중 `aria-busy`, 직접입력 input 에 placeholder + `aria-describedby`.
- 로딩 상태: 시군구/시스템 fetch 중 비활성 + "불러오는 중…" 옵션 1개. 실패 시 재시도 안내.
- ⚠ 판정: UI 변경이나 **신규 토큰/컴포넌트 없이 기존 폼 패턴 재사용**이라 디자인 리스크 낮음(A 정책 자문 완료, 토큰·다크모드·specificity 체크 통과).

## 7. API 설계

| 메서드/경로 | 입력 | 출력 |
|---|---|---|

## 8-1. 🗄 DB팀 가상 자문 결과
- additive nullable ALTER → **안전**(기존 데이터/제약 무영향).
- ⚠ `region_code` ↔ `sigungu_code.adm_sect_c` **FK 를 걸지 여부**: 미계약 데이터 유연성 위해 **FK 미설정 + 앱 검증(FR-11)** 권고(코드 마스터가 갱신될 수 있어 hard FK 는 입력 막을 위험).
- ⚠ **`Document.region_code` 선례와 차이 명시**(codex): 저장 *철학*(사업/인프라 독립 지역식별)은 동일하나, Document 쪽은 FK 적용 경로가 있고 본 건은 **FK 미설정+앱검증**으로 **제약 방식이 다름**. 개발계획서에 이 차이를 명기.
- ✅ `Infra.dist_cd` 점검 완료(2026-06-11): 205건 중 코드매칭 28(14%)·이름매칭 197(96%) → **이름 매칭 1차 키 확정**(§5-3).
- 통계 집계 대비 `tb_work_plan(region_code)` 인덱스 1개 권고.
