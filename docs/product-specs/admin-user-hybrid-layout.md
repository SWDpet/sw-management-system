## 1. 배경 / 목표

### 배경
`admin-user-list.html` 은 현재 **가로 16 컬럼 + 권한 10 select** 로 구성된 와이드 테이블. 실사용 시:
- 브라우저 화면을 꽉 채워도 개별 입력칸이 좁아 내용 잘림 (연락처·이메일·주소 등)
- 가로 스크롤 필수 → 여러 사용자 한눈 비교 어려움
- **민감정보(주민번호/연락처/주소) 항상 노출** → 보안·개인정보 관점 불리
- 권한 10개 select 가 테이블 가로 공간의 대부분 점유

2026-04-19 사용자 & codex 논의 결과: **하이브리드(목록 + 상세 확장) 레이아웃** 최적 → 본 기획서로 정식화.

### 목표

|---|------|-----------|

## 2. 사용자 시나리오

### 주 시나리오 1 — 신규 사용자 승인
1. 관리자 페이지 접속 → **가입 대기 목록** (기존 구조 유지) 에서 신규 승인 처리
2. 본 스프린트에서는 **가입 대기 테이블은 변경 없음** — 활성 사용자 관리 부분만 개편

### 주 시나리오 2 — 활성 사용자 권한 수정
1. 페이지 하단 **활성 사용자 목록**에서 대상 사용자 탐색 (아이디/성명/소속/권한 요약)
2. **[펼치기 ▾]** 버튼 클릭 → 해당 행 아래 상세 편집 영역 확장
3. 상세 영역에서 4 섹션 확인:
   - **기본정보** (아이디·성명 읽기 전용)
   - **소속·기술** (부서·팀·기술등급·업무 편집)
   - **민감정보** (마스킹, [🔒 보기] 버튼 클릭 시 평문 표시 + 편집 가능)
   - **권한** (10개 select 를 3~4열 그리드로 배치)
4. **[저장]** 클릭 → 기존 `<form>` POST `/admin/users/update/{id}` 제출 → 성공 시 리다이렉트 (서버가 `expand={id1,id2,...}` 쿼리 유지 후 redirect). 리로드 후 펼침 상태 복원.

### 주 시나리오 3 — 사용자 검색·비교
1. 기존 검색 폼 (타입/키워드) 그대로 유지
2. 검색 결과 목록에서 **권한 요약 뱃지** 로 "누가 어떤 권한 있는지" 한눈 비교
3. 상세 수정 필요 시에만 펼치기

### 주 시나리오 4 — 사용자 삭제
1. 목록 행의 **[삭제]** 버튼 (기존과 동일)
2. confirm 후 DELETE

| FR-2-G | 취소 시 상세 영역 **원본 값으로 복원** (서버 값 재로드 or 캐시된 초기값 복원) |

### 3-3. 민감정보 마스킹 (codex 지적 반영 — v2)

| ID | 내용 |
|----|------|
| FR-3-A | **서버 렌더 HTML 에 평문 포함 금지**. 페이지 초기 로드 시 민감 필드(`ssn`, `tel`, `mobile`, `email`, `address`) 는 **마스킹된 값만** DOM 에 존재. 평문은 `data-*` 속성에도 저장 금지. |
| FR-3-A-포맷 | 마스킹 포맷: `주민번호 901201-1******`, `연락처/휴대전화 010-****-1234`, `이메일 u***@domain.com`, `주소 서울 강남구 ****` (시군구까지만). |
| FR-3-B | 각 민감 필드 옆 **[🔒 보기]** 버튼. 클릭 시 서버 API 호출 `GET /admin/api/users/{id}/sensitive?field={ssn|tel|mobile|email|address}` → 응답 받은 평문을 **JS 변수(페이지 메모리)에만 보관** + 해당 input value 교체. 재클릭 시 JS 변수에서 제거 + DOM 마스킹 복원. |
| FR-3-C | 서버 엔드포인트 `/admin/api/users/{id}/sensitive`:
- ADMIN 권한 필수 (기존 `/admin/**` 체인)
- CSRF 면제 (기존 SecurityConfig `/admin/api/**` 이미 등록)
- 감사 로그 기록 (`LogService.log("사용자관리", "민감정보조회", "userSeq=X field=ssn")`)
- 응답 시 **필드 1개만** 반환 (한꺼번에 전체 수집 차단)
- 응답은 `{field: "ssn", value: "901201-1234567"}` 형태 |
| FR-3-D | 저장 form 제출 시 **편집 중 값은 input 에 보유** — 사용자가 평문을 보고 수정하면 form 에 수정된 값이 담겨 서버로 전송. 마스킹 상태에서는 편집 차단(input readonly). 편집하려면 반드시 "보기" 먼저 눌러 서버 확인. |
| FR-3-E | "보기" 상태로 전환 중에는 해당 필드 주변에 경고 표시 (예: "⚠ 평문 표시 중 — 편집 후 저장하세요"). 타 화면 이동 시 자동 재마스킹. |

## 5. 의사결정 / 우려사항

### 5-1. 상세 확장 = inline row vs 모달 — ✅ inline row 확장 확정
- 대안: 모달 팝업
- 선택: **inline row** — 여러 사용자 동시 펼쳐 비교 가능. 모달은 1명씩만 가능.

### 5-2. 권한 요약 뱃지 5개 초과 처리 — ✅ 앞 4 + "+N"
- 모든 뱃지 다 표시하면 목록 행 높이 증가 → 4개 + 요약. 상세 펼치면 전체 확인.

### 5-3. 민감정보 마스킹 전략 — ✅ v2: 서버 렌더에 평문 없음 + 개별 조회 API
- **v1 에서 `data-raw` 평문 저장 설계** 는 codex 지적으로 폐기.
- 초기 렌더: 서버가 마스킹된 값만 HTML 에 포함. 평문은 DOM·data attribute 어디에도 없음.
- "보기" 클릭: `/admin/api/users/{id}/sensitive?field=ssn` 호출 → 응답 평문을 JS 변수에만 저장 + 해당 input 값 교체. 감사 로그 자동 기록.
- 편집 권한: 마스킹 상태 = readonly, 보기 상태 = 편집 가능.
- 세션 탈취·확장프로그램 시나리오에서도 "클릭해야 조회" 이므로 일괄 수집 차단.

### 5-4. 저장 방식 — ✅ 기존 form 재사용 (form POST + redirect) 일관
- HTML `<form>` POST + 서버 redirect 방식으로 **통일** (v1 의 "PATCH" 언급은 v2에서 제거).
- form submit 시 펼침 상태는 hidden input `expand=1,2,3` 쉼표 직렬화로 전달.
- 서버는 update 처리 후 `redirect:/admin/users?page={page}&expand={ids}` 로 리다이렉트.
- AJAX·JS 상태저장 방식은 본 스프린트 범위 외.

### 5-5. 반응형 기준점 — ✅ 1280px, **1열 스택** 로 통일 (codex 지적 반영 v2)
- 1280px 이상: 상세 영역 2x2 섹션 그리드 (기본·소속 / 민감·권한)
- 1280px 미만: **1열 스택** 으로 전환. "카드형" 표현은 사용하지 않음.
- 목록 행은 1280px 이하에서도 테이블 유지 (가로 스크롤 허용). 테이블을 카드로 전환하지 않음.
- 모바일(768px 미만) 본 스프린트 **고려 범위 외** — 관리자 화면은 데스크톱 전제

### 5-6. 일괄 선택·승인 — ✅ 본 범위 외
- 복수 체크박스 + 일괄 액션 UI 는 별도 스프린트로 분리. 본 스프린트는 레이아웃 전환 집중.

## 8. 개발계획서 단계에서 재확인 필요 항목

1. 기존 update form 의 정확한 필드 목록 (변경 없음 가정이지만 재검증)
2. 민감정보 마스킹 함수의 경계 케이스 (빈 값·null·잘못된 포맷)
3. 권한 요약 뱃지 5개 초과 기준의 UX 마무리 (5개까지 보이고 +1 부터 요약, 또는 4개+요약)
4. `<form>` 태그 위치 — 현 구조가 `<form th:attr="id='updateForm'+...">` 방식이면 펼침 영역의 입력에 `form` 속성으로 연결
