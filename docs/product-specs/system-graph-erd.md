## 1. 배경 / 목표

### 배경
현재 ERD 는 두 가지 형태로만 제공됨:
- `docs/erd-*.png` — 정적 이미지 5장 (확대/검색/클릭 불가)
- `docs/erd-*.mmd` — Mermaid 소스 (개발자용, 일반 사용자는 렌더링 과정 필요)
- `docs/erd-diagram.html` — 정적 HTML (어떤 상호작용도 없음)

관리자/개발자가 특정 테이블을 찾거나 FK 관계를 추적할 때마다 5장 이미지를 열어 눈으로 찾아야 하는 불편이 있다.

### 목표
- `/admin/system-graph` 의 **C. ERD 인터랙티브** 탭에서 30+ 테이블을 하나의 네트워크 그래프로 탐색.
- **클릭 / 확대·축소 / 드래그 / 검색 / 도메인 필터** 제공.
- 데이터 출처는 **기존 `docs/erd-*.mmd` 파일 5개를 서버 기동 시 파싱**하여 JSON 캐시 (DB 변경 영향 없음).
- 향후 DB 스키마 실시간 인트로스펙션은 별도 Phase 로 미룸.

## 2. 사용자 시나리오

1. 관리자가 `/admin/system-graph` 접속 → **C. ERD 인터랙티브** 탭 클릭.
2. 5 개 도메인 (Infra / Contract / Document / Quotation / Core) 테이블 30+ 개가 네트워크 그래프로 한 번에 표시됨.
3. **도메인별 색상**으로 구분 (예: Infra=파랑, Contract=주황, ...).
4. 노드(테이블)를 클릭하면 **우측 상세 패널**에 해당 테이블의 컬럼 목록(PK/FK 표시 포함) 표시.
5. 상단 **검색창**에 테이블명 일부 입력 → 매치되는 노드 강조, 나머지 흐리게.
6. 상단 **도메인 필터 체크박스**로 특정 도메인만 표시 가능.
7. 엣지(FK 관계) 에 마우스 호버 → 관계 레이블 표시 (예: "has servers").
8. 마우스 휠로 확대/축소, 드래그로 팬.

## 3. 화면 / UI 흐름

### 3-1. 기본 레이아웃

```
┌──────────────────────────────────────────────────────────────┐
│ [🔍 검색: ______] [✓ Infra] [✓ Contract] [✓ Document] ...    │ ← 필터 바
├────────────────────────────────────┬─────────────────────────┤
│                                    │ 선택된 테이블 상세      │
│   네트워크 그래프 (vis-network)    │ ─────────────────────   │
│                                    │ tb_infra_server         │
│   ● ── ● ── ●                      │ ─────────────────────   │
│   │     │                          │ 🔑 server_id (PK)       │
│   ● ── ●                           │ 🔗 infra_id (FK)        │
│                                    │     → tb_infra_master   │
│   (zoom/pan/drag 가능)             │    server_type          │
│                                    │    ip_addr              │
│                                    │    os_nm                │
│                                    │    ...                  │
└────────────────────────────────────┴─────────────────────────┘
```

### 3-2. 색상 팔레트 (도메인별)

| 도메인 | 색상 | 코드 |
|--------|------|------|
| Infra | 파랑 | `#4e73df` |
| Contract | 주황 | `#f6c23e` |
| Document | 초록 | `#1cc88a` |
| Quotation | 자주 | `#a14ed0` |
| Core | 회색 | `#858796` |

### 3-3. 노드 속성
- 모양: `box` (rounded)
- 라벨: 테이블명 (예: `tb_infra_server`)
- 툴팁(hover): 테이블 설명(있으면) + 컬럼 개수
- 클릭: 상세 패널에 컬럼 목록 표시

### 3-4. 엣지 속성
- 화살표: 방향 있음 (PK → FK)
- 라벨: .mmd 에 있는 관계 설명 (예: "has servers")
- 스타일: 실선, 호버 시 굵게

### 3-5. 검색 동작
- 입력 즉시(타이핑 각 글자마다) 매칭되는 노드 **강조(하이라이트)** + 나머지 **opacity 0.25**.
- 검색어 비우면 모두 정상 표시.

### 3-6. 필터 동작
- 체크박스 해제하면 해당 도메인 노드들이 숨겨짐 (엣지도 자동 숨김).
- 모든 도메인 해제하면 "표시할 테이블이 없습니다" 안내 문구.

## 4. 기능 요건 (FR)

| ID | 내용 |
|----|------|
| FR-1 | 서버 기동 시 `docs/erd-*.mmd` 5개 파일을 읽어 테이블·컬럼·관계를 파싱하고 메모리 캐시에 저장. 중복 테이블은 **FR-1-MERGE 규칙**에 따라 정본 1개만 남긴다. |
| FR-1-MERGE | **중복 테이블 병합 규칙**: 같은 테이블명이 여러 `.mmd` 에 등장하면, **컬럼 수가 가장 많은 정의를 정본**으로 선택. 컬럼 수 동률이면 도메인 우선순위 **`core > infra > contract > document > quotation`** 로 결정. 스텁(부분) 정의는 버리되, 관계선은 모든 파일의 합집합을 엣지로 사용(from-to 쌍 기준 중복 제거). 선택된 정본의 원천 파일이 노드의 도메인(=색상)이 된다. |
| FR-2 | `GET /admin/system-graph/api/erd` 엔드포인트가 캐시된 JSON (nodes, edges) 을 반환. 응답 형식은 아래 "JSON 스키마" 참고. |
| FR-3 | C 탭 활성화 시 해당 API 호출 → vis-network 렌더링. 재탭 전환 시 캐시 재사용 (재요청 없음). |
| FR-4 | 노드 클릭 시 우측 상세 패널에 컬럼 목록 (컬럼명 / 타입 / PK/FK 배지) 표시. |
| FR-5 | FK 컬럼 중 **관계선(`||--o{` 등)에 실제 등장하는 FK** 에 한해 클릭 시 참조 테이블로 **포커스 이동**(해당 노드 선택 + 카메라 중앙 배치). 관계선에 없는 FK 컬럼(예: `tb_document.contract_id` 처럼 .mmd 에 양방향 매핑이 누락된 경우) 은 **FK 배지만 표시하고 점프 불가**, 호버 툴팁에 "관계선 미정의로 이동 불가" 안내. |
| FR-5-A | `fkRef` 필드는 양방향 관계선이 .mmd 에 존재하는 FK 에 대해서만 채워진다. 없으면 `null`. 프론트는 `fkRef == null` 일 때 클릭을 비활성화한다. |
| FR-6 | 검색창: 테이블명 부분일치(대소문자 무시) 매칭 노드 강조, 기타 노드 흐리게. |
| FR-7 | 도메인 필터: 체크박스 5개, 초기 상태 모두 체크. 해제 시 해당 도메인 노드/엣지 즉시 숨김. |
| FR-8 | 엣지 관계 레이블(.mmd 에서 파싱한 설명 문자열)은 **기본 숨김 + 마우스 호버 시 표시** — 사용자 시나리오·리스크 완화책과 일치. 항상 표시 옵션 없음. |
| FR-9 | 도메인별 색상 구분 (섹션 3-2 팔레트). |
| FR-10 | 확대·축소(wheel), 팬(drag), 노드 드래그 지원. |
| FR-11 | 초기 로드 시 물리 시뮬레이션으로 자동 배치(`physics.stabilize`) 후 정지 — 사용자 조작 전까지 안정 상태 유지. |
| FR-12 | API 응답이 비어 있거나 오류 시 "ERD 데이터를 불러오지 못했습니다. server.log 확인 필요" 안내. |

## 7. 의사결정 / 우려사항

### 7-1. 데이터 소스 — ✅ 확정
- **선택**: `docs/erd-*.mmd` 파일 파싱
- 근거: 현존 자료 재사용, DB 비종속, 구현 단순

### 7-1-A. 중복 테이블 병합 규칙 — ✅ 확정 (v2)
- 컬럼 수 최대 정의를 정본으로 채택. 동률 시 도메인 우선순위 `core > infra > contract > document > quotation`.
- 확인 사례:
  - `users` → `erd-core.mmd` (8컬럼) 정본, `erd-contract.mmd` 의 4컬럼 스텁은 버림. 도메인=Core.
  - `tb_infra_master` → `erd-infra.mmd` (6컬럼) 정본, `erd-contract.mmd` 의 2컬럼 스텁은 버림. 도메인=Infra.
- 엣지는 모든 파일의 관계선 합집합 (from-to 기준 중복 제거).

### 7-1-B. FK 참조 추론 — ✅ 확정 (v2)
- `.mmd` 관계선에 존재하는 FK 만 `fkRef` 채움. 나머지는 `null`.
- 클릭 네비게이션은 `fkRef != null` 인 경우에만 활성.
- 알려진 한계: `tb_document` 처럼 다중 FK 컬럼이 양방향 관계선으로 명시되지 않은 경우 일부 점프 불가 — Phase 2 에서 DB information_schema 보완 고려.

### 7-1-C. 엣지 레이블 표시 규칙 — ✅ 확정 (v2)
- **기본 숨김 + 호버 시 표시** 로 단일화. FR-8/시나리오/리스크 완화책 모두 일치.

### 7-2. 파싱 방식 — 개발계획서에서 세부 결정
- 후보 A: 정규식 기반 경량 파서 (권장, 외부 의존성 없음)
- 후보 B: Mermaid 공식 JS 파서 포트 (JVM 쪽 라이브러리 부재로 복잡)
- 개발팀에서 후보 A 를 기본 채택.

### 7-3. 캐시 갱신 정책 — 확정
- 서버 재기동 시 1회 파싱 후 메모리 보관.
- 운영 중 .mmd 업데이트 시 재기동 필요 (현재 운영 패턴과 일치).

### 7-4. 향후 확장
- 테이블 상세 패널에 "실제 DB 컬럼 대조" 버튼 추가 → information_schema 조회로 정합성 체크 (Phase 2 이후).
- 노드 클러스터링(대규모 ERD 시) — 현재 30 개 규모라 불필요.
