기획서 — 유지보수유형(maint_type) 기반 점검범위 분기

작성: 2026-05-30 · 워크플로우: 요청 → 기획서(본문서) → DB/디자인팀 자문 → ~~codex 검토(속도모드 생략)~~ → 사용자 최종승인 → 개발계획 → 구현 → 전체검증 합의 출처: 2026-05-30 사용자 협의 (maint_type 5종 / 점검 프로파일 3종 / sys_nm 분리 확정)

1. 배경 / 문제

점검내역서는 현재 모든 사업을 전체 섹션(AP·DB·DBMS·GIS·표준시스템) 가정으로 다룬다. 그러나 실제 사업은 유지보수 범위가 다르다:

  • 통합(HW+SW) 사업: 운영장비(AP/DB) + DBMS + GIS + 표준시스템 — 예: 강진군 UPIS
  • SW 유지보수: 자사 SW(GIS엔진)만, 또는 GIS + 표준시스템
  • 시스템마다 표준시스템 보유 여부가 다름: UPIS는 보유, KRAS는 GIS엔진만

→ SW 사업인데 빈 AP/DB 섹션이 "수집 대기"로 뜨면 보는 사람이 "누락인가?" 혼란. 점검 범위를 사업 유형에 맞춰 줄이고(lean), 어떤 유형인지 직관적으로 보이게 해야 한다.

2. 목표

  1. maint_type → 점검 범위 단일 규칙으로 섹션을 자동 분기 (lean)
  2. 리포트/목록/작성UI에 유형을 직관적으로 표출 (배지·범위 칩)
  3. 새 시스템/유형 추가에 확장 가능한 구조 (코드 조합 폭발 없음)

3. 확정 설계

3-1. maint_type 5종 (구성: D=DB구축 · H=HW운영장비 · S=SW(GIS엔진))

코드 유형명(nm) 사업 구성 점검 프로파일 상태
S SW GIS엔진 GIS만 🆕 신규
SU SW/표준 GIS엔진 + 표준시스템 GIS+표준 🆕 신규
DS DB/SW DB구축 + GIS엔진 GIS만 기존
HS HW/SW AP·DB·DBMS·GIS·표준 전체 기존
DHS DB/HW/SW DB구축 + AP·DB·DBMS·GIS·표준 전체 기존
  • SU의 U = 표준시스템(일반) — 시스템 구분은 sys_nm_en이 담당 (UPIS표준/KRAS… ). 새 시스템도 SU 코드 재사용.
  • DB구축(D)은 산출물이지 점검 섹션이 아님 → 점검상 S=DS, HS=DHS.

3-2. 점검 프로파일 3종

프로파일 maint_type AP/DB(HW) DBMS GIS 표준시스템
GIS만 S, DS ✗ ✗ ✅ ✗
GIS+표준 SU ✗ ✗ ✅ ✅
전체 HS, DHS ✅ ✅ ✅ ✅

3-3. 단일 도출 규칙

HW(AP/DB) · DBMS 점검  ⟺  maint_type ∈ {HS, DHS}
표준시스템 점검         ⟺  maint_type ∈ {SU, HS, DHS}  AND  sys_nm 이 표준시스템 보유
GIS엔진 점검           ⟺  항상
점검 항목 내용         =  sys_nm_en (UPIS→GeoNURIS+UPIS표준, KRAS→KRAS GIS …)

3-4. 시스템 × maint_type 유효 조합 (예시)

시스템 표준시스템 가능한 maint_type
UPIS 보유 S · SU · DS · HS · DHS
KRAS 없음(GIS만) S (· DS)

→ 시스템이 프로파일을 제약. 표준시스템 없는 시스템은 SU/HS/DHS의 표준 섹션이 자동 비활성.

4. 컴포넌트별 변경

# 영역 변경
1 코드마스터 maint_tp_mst 에 S(SW), SU(SW/표준) INSERT (db_init seed)
2 점검범위 유틸 maint_type → Set<Section> 매핑 유틸 (공통). 리포트·작성UI·집계 공용. sys_nm 표준보유 여부 반영
3 리포트 v2 ① 표지 정보표에 유형 배지(SW/GIS·HW/SW…) ② 요약 대시보드에 점검 범위 칩 ③ 범위 밖 섹션은 "수집 대기" 아닌 비표시(lean) — 이미 데이터주도라 카드 자동 생략, "기대섹션" 판정만 프로파일 반영
4 작성 UI(doc-inspect) 유형(프로파일)에 따라 범위 밖 섹션 입력 탭/칸 숨김 (SW면 AP/DB/DBMS 숨김, GIS만이면 표준시스템도 숨김)
5 운영문서 목록 유형 컬럼/배지 (시도·시군구 정리에 이어)
6 QR 자동생성/적재 report 생성 시 사업 maint_type 읽어 프로파일 결정 (InspectionQrBatchService / InspectReportService)
7 에이전트(폐쇄망, 별도 라운드) site.{code}.json 에 sections/profile 추가 → Collector가 범위 밖 모듈 미실행 (SW사이트는 AP/DB 접근 시도 안 함)

5. 직관적 표출 (디자인 방향)

  • 표지 유형 배지: 색+라벨 — 예) SW · GIS엔진(티일) / HW/SW · 운영장비(네이비) / SW/표준(앰버)
  • 요약 대시보드 범위 칩: "점검 범위: GIS엔진 + 표준시스템" → 빈 섹션이 누락 아님을 명시
  • GIS만 리포트 = 표지 배지 + GIS 카드 + (해당 시) 표준 카드만 → 한눈에 인지
  • 디자인팀 자문 항목: 배지 색/문구 체계, 범위 칩 위치

6. 단계별 추진

  1. 데이터/코드: maint_tp_mst S·SU 추가 + 점검범위 매핑 유틸 + 기존 사업 maint_type 점검(강진=HS/DHS 확인)
  2. 리포트 v2: 유형 배지 + 범위 칩 + 프로파일 기반 기대섹션
  3. 작성 UI: 유형별 섹션 토글
  4. 운영문서 목록: 유형 배지
  5. 에이전트(폐쇄망 별도): site config sections 필터

7. 마이그레이션 / 호환

  • 기존 사업: maint_type 값 그대로. S/SU 신설은 추가 INSERT(비파괴).
  • maint_type 미지정(null) 사업 → 기본 전체 프로파일(보수적)로 폴백.
  • 강진군(샘플) = HS 또는 DHS (확인 후 확정).

8. 미결 / 확인 필요

  • S/SU 표시 문구(nm) 최종 확정: "SW" / "SW/표준" (또는 다른 표기)
  • 표준시스템 보유 시스템 마스터: sys_nm 별 "표준 보유 여부"를 어디서 판단할지 — 표준시스템 템플릿(inspect_template) 존재 여부로 판단 vs sys_mst 플래그 추가
  • 강진군 실제 maint_type 값 확인

9. 검증 계획 (속도모드 — 개발 완료 후 일괄)

  1. 유형별(S/SU/DS/HS/DHS) 리포트 PDF 생성 → 섹션·배지·칩 육안(PDFBox 래스터)
  2. 작성 UI 유형 전환 시 섹션 토글
  3. QR 적재 시 프로파일 자동결정
  4. 기존 전체 사업(강진) 회귀 없음
